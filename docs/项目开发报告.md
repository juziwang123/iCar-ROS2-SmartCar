# iCar 智能小车 ROS2 项目开发报告

## 1. 项目概况

### 1.1 项目名称

基于 iCar 智能小车的 ROS2 多传感器巡检与自主导航系统。

### 1.2 开发目标

围绕实训任务，完成智能小车在远程连接、底盘控制、激光雷达感知、视觉检测、SLAM 建图、自主导航和上位机控制方面的系统化开发，并构建可持续维护的仓库结构、文档体系和测试流程。

### 1.3 开发原则

1. 先保证必做任务闭环，再评估进阶功能。
2. 先打通最短可运行链路，再逐步增加模块复杂度。
3. 文档、代码和测试同步推进，避免阶段末集中补材料。

### 2.3 当前仓库状态

截至 7 月 7 日，仓库已完成以下工作：

5. 代码已推送至 GitHub，Python 3.8 兼容

### 2.4 小车已有能力确认

通过阅读《智能小车使用手册》，确认小车预置以下功能（`icar_*` / `yahboomcar_*` 官方包）：

| 功能 | 容器 | 指令 |
|------|------|------|
| 底盘驱动 | iCar | `n1` |
| 雷达驱动 | iCar | `n2` |
| 雷达避障 | iCar | `n3` |
| 雷达跟随 | iCar | `n4` |
| 雷达警卫 | iCar | `n5` |
| 深度相机 | iCar | `m1` |
| HSV 颜色识别 | iCar | `m2` |
| 颜色追踪 | iCar | `m3` |
| gmapping 建图 | 导航容器 | `m1`-`m4` |
| DWA/TEB 导航 | 导航容器 | `n1`-`n4` |

因此我们的开发策略调整为：不重复实现小车已有功能，而是在其之上构建控制增强层（safety_mux + motion_controller），将多个独立功能统一仲裁。`car_lidar/avoidance.py` 和 `car_lidar/tracker.py` 作为备选方案，可按需替换小车官方节点。

## 3. 系统开发方案

### 3.1 总体架构

系统以 ROS2 为通信中枢，围绕 /cmd_vel、/scan、/map、图像流等核心话题构建。整体开发划分为以下层次：

1. 设备与环境层：小车底盘、雷达、相机、远程连接。
2. 驱动与接入层：底盘控制、传感器驱动、基础话题输出。
3. 算法与业务层：避障、跟随、建图、导航、目标检测。
4. 交互与展示层：APP 控制、远程桌面、演示材料。
5. 工程支撑层：脚本、测试、文档、版本管理。

### 3.2 模块划分与小车已有功能对照

| 我们的模块 | 定位 | 小车已有对应功能 |
|-----------|------|-----------------|
| car_bringup | 统一启动与参数管理 | —（新增层） |
| car_control | 控制仲裁、急停、模式切换 | —（新增层） |
| car_lidar | 避障/跟随（备选方案） | icar_laser（n3/n4/n5） |
| car_vision | YOLO 检测等进阶视觉 | icar_astra（m2/m3） |
| car_navigation | goal_publisher / 巡航 | yahboomcar_nav（m1-m4/n1-n4） |
| car_app_bridge | APP/LLM 桥接 | —（新增层） |

## 4. 开发过程说明

### 4.1 第一阶段：需求梳理与环境准备

本阶段主要工作包括：

1. 汇总课程要求、答辩节点与提交物清单。
2. 阅读小车连接指南，明确 SSH、NoMachine、VNC 等接入方式。
3. 确定本地开发环境与实车部署环境的版本差异。
4. 建立 GitHub 仓库并组织初始目录结构。

阶段产出：

1. 项目汇总材料。
2. README 初稿。
3. 本报告及相关正式文档。

### 4.2 第二阶段：系统骨架搭建

本阶段主要工作包括：

1. 依据功能边界划分 ROS2 包目录。
2. 预留 launch、config、models、maps、tests 等子目录。
3. 统一命名规范，为后续模块填充提供稳定结构。

本阶段目标不是一次性完成所有代码，而是先把系统组织方式定下来，降低多人并行开发时的冲突和返工风险。

### 4.3 第三阶段：核心功能实现

计划按以下顺序推进：

1. 底盘控制：完成速度控制、方向控制、手动遥控。
2. 雷达功能：完成雷达数据订阅、距离阈值判断和基础避障。
3. 建图导航：完成地图构建、目标点发布和导航验证。
4. 视觉模块：完成颜色检测，再迭代到 YOLO 识别。
5. 上位机桥接：完成外部指令到 ROS2 的转换。

### 4.4 第四阶段：联调与演示优化

本阶段主要聚焦以下事项：

1. 控制优先级梳理，避免手动控制与自动控制互相覆盖。
2. 参数调优，平衡速度、检测阈值和导航稳定性。
3. 演示流程固化，保证答辩现场可重复操作。
4. 同步补齐测试记录、日志和截图素材。

## 6. 关键技术问题与处理思路

### 6.1 ROS2 版本差异问题

问题：本地使用 Humble，实车可能使用 Foxy，接口行为和依赖包安装方式可能不同。

处理思路：

1. 本地开发尽量使用标准消息和标准包接口。
2. 部署前记录差异项，单独适配 Launch、依赖与运行命令。
3. 将环境说明写入 README 与使用手册，避免误用同一工作空间。

### 6.2 实车调试资源受限问题

问题：实训期间设备共享，联调时间有限。

处理思路：

1. 尽量先在本地完成逻辑与仿真验证。
2. 实车调试前准备好命令、脚本、参数和回滚方案。
3. 将调试过程中的问题沉淀为故障排查清单。

### 6.3 演示稳定性问题

问题：答辩环境复杂，网络、光照、障碍物布局都会影响展示效果。

处理思路：

1. 选择稳定场景和固定展示路线。
2. 保留手动接管与急停手段。
3. 预准备视频、截图和替代演示方案。
