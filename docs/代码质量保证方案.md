# iCar 智能小车 ROS2 项目代码质量保证方案

## 1. 文档目的

本文档用于规范 iCar 智能小车 ROS2 项目的代码质量保证方式，明确开发过程中的质量目标、控制手段、检查流程和责任分工，确保项目在实训周期较短的情况下，仍能保持代码可读、可测、可维护、可演示。

## 2. 适用范围

本方案适用于以下内容：

1. ROS2 功能包源码。
2. Launch、配置、脚本等工程辅助文件。
3. 测试脚本与测试记录。
4. README 与核心技术文档中涉及实现说明的部分。

## 3. 质量目标

### 3.1 基本目标

1. 代码结构清晰，模块边界明确。
2. 关键功能具备最小可验证链路。
3. 提交内容与文档描述一致。
4. 出现问题时能够较快定位到模块和原因。

### 3.2 项目级质量要求

1. 所有核心模块必须有明确负责人。
2. 所有核心功能必须至少经过一次独立验证。
3. 所有重要变更必须记录到仓库并保留说明。
4. 所有答辩演示功能必须具备回退方案或替代展示材料。

## 4. 代码质量标准

### 4.1 结构规范

1. 按功能包组织代码，不把多个模块逻辑混写在单个文件中。
2. 启动文件、配置文件、源码、模型、地图等资源应分类存放。
3. 每个功能包应保持职责单一，避免控制、感知、导航逻辑强耦合。
4. 目录结构必须与 README 和文档说明一致。

### 4.2 可读性规范

1. 变量名、函数名、节点名应具备明确语义。
2. 不使用难以理解的缩写或临时命名。
3. 单个函数应尽量只完成一种职责。
4. 复杂逻辑应拆分成多个步骤，必要时补充简洁说明。

### 4.3 可维护性规范

1. 阈值、速度、路径、模型文件等可变内容优先配置化。
2. 不将机器相关路径、账号信息、临时参数直接写死到核心逻辑中。
3. 调试代码、临时输出和无用文件在提交前必须清理。
4. 模块接口应尽量使用标准 ROS2 消息类型，减少后续适配成本。

### 4.4 稳定性规范

1. 节点启动失败时应输出明确错误信息。
2. 涉及小车运动的代码必须优先保证停止和急停逻辑可靠。
3. 自动控制逻辑必须考虑控制优先级，避免多个模块同时争抢底盘控制。
4. 传感器数据异常时应优先降级处理，而不是直接导致整条链路崩溃。

## 5. 开发过程质量控制

### 5.1 开发前检查

每次开始开发前应确认：

1. 当前任务负责人明确。
2. 修改目标模块明确。
3. 本次改动对应的验证方式明确。
4. 是否会影响其他模块已有演示链路。

### 5.2 开发中要求

1. 尽量以小步提交方式推进，避免一次性大改多个模块。
2. 单次改动应围绕一个明确目标，例如修复控制链路、增加避障判断或补充启动参数。
3. 开发完成后立即做最小验证，不把未验证代码直接并入主线说明材料。
4. 若实车不可用，应先完成本地逻辑验证或命令级验证。

### 5.3 开发后检查

每次提交前至少确认以下事项：

1. 代码能被正常读取和理解。
2. 本次修改不包含明显无关变更。
3. 关键命令、参数、路径和文件名无拼写错误。
4. 文档中涉及该功能的描述已同步更新。

## 6. 测试与验证要求

### 6.1 最小验证原则

每个核心模块都必须有最小验证动作：

1. 控制模块：验证控制命令能到达底盘或控制接口。
2. 雷达模块：验证 /scan 数据可读取，避障逻辑能触发。
3. 视觉模块：验证图像流可获取，检测结果可输出。
4. 导航模块：验证建图、目标点发布或地图加载中的至少一项。

### 6.2 集成验证原则

集成验证时重点检查：

1. 话题名称是否一致。
2. 消息类型是否匹配。
3. 自动控制与手动控制是否冲突。
4. Launch 文件是否能按预期启动相关节点。

### 6.3 演示前验证原则

答辩或演示前必须完成：

1. 至少一次完整流程彩排。
2. 至少一次急停或停止逻辑验证。
3. 网络、供电、设备连接状态确认。
4. 备用视频、截图或日志证据准备。

## 7. 代码评审要求

### 7.1 评审范围

以下改动建议经过至少一名组员复核：

1. 底盘运动控制逻辑。
2. 避障和导航控制策略。
3. Launch 启动入口和公共配置文件。
4. 会影响演示主链路的集成改动。

### 7.2 评审检查点

评审时至少检查以下内容：

1. 代码是否实现了声明的目标。
2. 是否引入了明显的控制风险或兼容性问题。
3. 是否修改了公共接口但没有同步文档。
4. 是否保留了临时调试代码或错误配置。

### 7.3 评审结论记录

建议采用简单记录方式：

| 日期 | 提交内容 | 复核人 | 结论 | 备注 |
|------|----------|--------|------|------|
|  |  |  | 通过/修改后通过/退回 |  |

## 8. 配置与依赖管理要求

1. 依赖项安装方式应在 README 或脚本中体现。
2. 参数文件应集中管理，避免相同参数散落多个位置。
3. 地图、模型、脚本等资源必须说明用途和适用环境。
4. 不将本地私有路径、临时缓存文件或构建产物纳入正式提交。

## 9. 文档一致性要求

1. README 中描述的目录结构应与实际仓库一致。
2. 文档中提到的模块名称应与代码目录命名一致。
3. 若代码尚未实现，文档中必须使用“计划实现”“待完成”“预留模块”等表述。
4. 测试报告和开发报告中的结果必须基于真实验证，不得虚构。

## 10. 角色责任划分

| 角色 | 质量责任 |
|------|----------|
| 组长 | 统筹质量要求、控制范围、审核最终版本 |
| 控制负责人 | 保证运动控制安全、接口清晰、启动可用 |
| 雷达/导航负责人 | 保证感知与自动控制逻辑可验证、可调试 |
| 视觉负责人 | 保证图像输入与识别结果可复现 |
| 测试与交付负责人 | 保证测试记录完整、文档与代码一致 |

## 11. 提交前质量检查清单

提交前逐项检查：

- 是否只提交了与当前任务相关的改动。
- 是否清理了调试输出、临时代码和无关文件。
- 是否完成了对应模块的最小验证。
- 是否更新了 README 或相关文档。
- 是否确认当前改动不会破坏演示主链路。
- 是否保留了必要的日志、截图或测试记录。

## 12. 当前项目适用说明

结合当前仓库现状，本项目当前重点不是追求复杂的自动化质量平台，而是先建立一套轻量但有效的人工质量保证机制。原因如下：

1. 仓库当前仍以功能骨架和文档体系为主。
2. 核心代码尚在逐步补充阶段，自动化测试覆盖率短期内不会很高。
3. 实车项目对“安全、可演示、可回退”的要求高于纯离线算法项目。

因此，现阶段最有效的质量保证方法是：

1. 小步开发。
2. 小步验证。
3. 文档同步。
4. 关键改动复核。
5. 演示前全链路彩排。

## 13. 结论

代码质量保证的核心不在于文档数量，而在于是否把“谁负责、怎么检查、何时验证、出现问题如何回退”说清楚并执行。对当前这个 ROS2 智能小车项目而言，只要按本方案落实最小验证、模块复核和交付检查，就能在有限时间内显著降低功能失效和答辩翻车风险。
