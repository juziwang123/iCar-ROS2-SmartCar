# iCar 智能小车 ROS2 项目提交规范与交付清单

## 1. 文档目的

本文档用于统一项目提交材料的内容范围、命名方式、版本管理方式、提交流程和验收检查项，避免在中期检查、最终答辩和结项提交时出现材料缺失、版本不一致或职责不清的问题。

## 2. 适用范围

本文档适用于以下提交场景：

1. 每日开发日报提交。
2. 中期检查材料提交。
3. 最终答辩材料提交。
4. 结项代码、文档、视频与使用说明提交。

## 3. 提交物清单

### 3.1 必交材料

| 序号 | 提交物 | 说明 | 截止时间 |
|------|--------|------|----------|
| 1 | 每名组员日报 | 每日记录工作内容、问题、计划 | 每天 |
| 2 | 需求分析报告 | 说明项目背景、需求、验收标准 | 答辩前 |
| 3 | 中期检查 PPT | 汇报阶段进度、已完成功能和问题 | 7 月 11 日 |
| 4 | 最终答辩 PPT | 汇报完整方案、实现效果和总结 | 7 月 15 日 |
| 5 | 项目开发报告 | 说明开发过程、环境、分工、问题处理 | 答辩后 2 日 |
| 6 | 项目测试报告 | 说明测试范围、用例、结果、问题 | 答辩后 2 日 |
| 7 | 项目使用手册 | 说明环境准备、连接、启动、使用、排障 | 答辩后 2 日 |
| 8 | 项目代码仓库 | GitHub 或 Gitee 仓库地址及源码 | 答辩后 2 日 |
| 9 | 演示视频 | 展示核心功能和操作流程 | 答辩后 2 日 |

### 3.2 建议补充材料

| 序号 | 提交物 | 说明 |
|------|--------|------|
| 1 | 设计文档 | 说明系统架构、模块设计、接口与流程 |
| 2 | 项目分工与任务清单 | 说明每名成员承担内容与完成情况 |
| 3 | 联调问题清单 | 记录典型问题、修复过程和剩余风险 |
| 4 | 运行截图或日志证据 | 作为测试和答辩补充材料 |

## 4. 目录组织规范

建议项目交付时按以下结构整理：

```text
项目根目录/
├── README.md
├── src/
├── docs/
│   ├── 需求分析报告.md
│   ├── 项目开发报告.md
│   ├── 设计文档.md
│   ├── 测试报告.md
│   ├── 使用手册.md
│   ├── 提交规范与交付清单.md
│   ├── 项目分工与任务清单.md
│   └── 日报/
├── tests/
└── scripts/
```

交付时应保证仓库目录、README 描述和实际文件一致。

## 5. 文件命名规范

### 5.1 文档命名规范

1. 正式文档统一使用中文名称，避免同一类材料出现多个版本文件名。
2. 如果需要区分阶段版本，建议使用“文件名_v1.0.md”形式。
3. 最终提交版建议去除临时后缀，仅保留正式名称。

推荐命名如下：

1. 需求分析报告.md
2. 项目开发报告.md
3. 设计文档.md
4. 测试报告.md
5. 使用手册.md
6. 提交规范与交付清单.md
7. 项目分工与任务清单.md

### 5.2 PPT 与视频命名规范

建议命名格式：

1. 第X组_中期检查汇报.pptx
2. 第X组_最终答辩汇报.pptx
3. 第X组_项目演示视频.mp4

### 5.3 日报命名规范

建议按日期统一命名，例如：

1. 7月6日_张三_日报.md
2. 7月7日_李四_日报.md

## 6. 内容编写规范

### 6.1 通用要求

1. 所有正式文档应保持标题、编号、术语和项目名称统一。
2. 文档中的功能描述必须与代码实现和演示结果一致。
3. 不得把计划中的内容写成已完成事实。
4. 文档中的截图、命令、路径和仓库链接应真实可复现。

### 6.2 需求分析报告要求

至少应包含：

1. 项目背景。
2. 目标与范围。
3. 功能需求与非功能需求。
4. 约束条件。
5. 风险与验收标准。

### 6.3 项目开发报告要求

至少应包含：

1. 开发环境与资源。
2. 系统开发方案。
3. 阶段过程记录。
4. 分工协作方式。
5. 问题与处理思路。

### 6.4 测试报告要求

至少应包含：

1. 测试目标与范围。
2. 测试环境。
3. 测试用例。
4. 测试结果。
5. 缺陷与剩余风险。

### 6.5 使用手册要求

至少应包含：

1. 环境准备。
2. 连接方式。
3. 启动流程。
4. 常用命令。
5. 故障排查与安全注意事项。

## 7. 版本管理规范

1. 所有正式文档统一放在 docs 目录下维护。
2. 重要节点提交前必须检查文档与代码版本是否一致。
3. 由组长负责合并最终提交版本，避免多人各自保存独立终稿。
4. 演示前一天冻结答辩版材料，非必要不再大改结构。

## 8. 提交流程规范

### 8.1 日常提交流程

1. 每名组员当天整理开发内容与问题。
2. 更新个人日报与负责模块文档。
3. 提交代码并同步给组长。
4. 组长汇总全组进展和风险。

### 8.2 中期检查提交流程

1. 汇总已完成模块与可演示功能。
2. 形成中期检查 PPT。
3. 准备至少一条稳定演示链路。
4. 检查仓库是否存在无法解释的空文档或失效链接。

### 8.3 最终答辩提交流程

1. 锁定最终功能清单与演示流程。
2. 更新 README、需求、开发、测试、手册等正式文档。
3. 检查 PPT、视频、截图和代码仓库一致性。
4. 完成答辩版归档。

### 8.4 结项提交流程

1. 将答辩后修订版文档整理为最终版。
2. 检查代码可访问性、文档完整性和视频可播放性。
3. 提交统一打包材料或统一仓库链接。

## 9. 提交前检查清单

### 9.1 文档检查

- 项目名称、组号、成员信息是否统一。
- 文档标题、章节编号和术语是否一致。
- 代码实现状态与文档描述是否一致。
- 仓库地址、命令、截图、流程是否可复现。
- 文档中是否仍保留“待填写”“后续补充”等未清理内容。

### 9.2 代码检查

- README 是否能说明项目用途和运行方式。
- 目录结构是否清晰。
- 临时文件、无关文件是否已排除。
- 核心代码和文档是否已经推送到统一仓库。

### 9.3 演示检查

- 演示场景是否已经实测。
- 急停与人工接管方案是否明确。
- 网络、供电、设备连接是否稳定。
- 备用视频、截图和讲解顺序是否准备完毕。

## 10. 责任建议

1. 组长负责最终版本收口、统一命名和提交。
2. 各模块负责人负责本模块代码、测试证据和文档原始内容。
3. PPT 负责人负责统一视觉风格和讲述逻辑。
4. 视频负责人负责演示录制、剪辑和文件归档。

## 11. 结论

规范性文档的核心作用不是增加材料数量，而是让全组在最后提交前有统一标准、有统一版本、有统一检查口径。只要严格按本清单推进，就能显著降低交付阶段的混乱和返工。
