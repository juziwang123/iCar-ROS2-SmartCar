# iCar 统一启动与测试指南

本指南面向 Jetson/Foxy 车端。先进入工作空间并加载环境；不要在 Windows 开发机直接执行 ROS 2 实车命令。

```bash
cd ~/ros2_ws
source /opt/ros/foxy/setup.bash
source install/setup.bash
```

## 1. 统一启动入口

重新构建 `car_bringup` 后，可使用唯一入口 `icar`：

```bash
colcon build --packages-select car_bringup car_mission car_control car_app_bridge
source install/setup.bash

# 仅建图（与导航/巡检互斥）
ros2 run car_bringup icar start mapping

# 导航定位
ros2 run car_bringup icar start navigation --map src/car_navigation/maps/lab_map.yaml --rviz

# 完整巡检；按现场实际路径传入 map 和 route
ros2 run car_bringup icar start mission \
  --map src/car_navigation/maps/lab_map.yaml \
  --route src/car_mission/config/demo_inspection_route.yaml \
  --app-bridge --yolo
```

启动前可用 `--dry-run` 查看固定的 `ros2 launch` 参数。入口只提供三种经过 `bringup.launch.py` 互斥规则校验的 profile，不拼接 shell 字符串，也不会绕过 `safety_mux`。

## 2. 交互式任务控制台

在系统已经启动的第二个终端执行：

```bash
source ~/ros2_ws/install/setup.bash
ros2 run car_bringup icar console
```

控制台是车端本地驾驶员界面：手动速度只发布到 `/cmd_vel_manual`，仍由 `safety_mux` 仲裁；它不允许任意 Topic 或 Shell 命令。不要同时让 APP 与控制台争夺手动控制权。

```text
status
mode manual
move 0.15 0.0
stop
mode nav
nav 1.2 0.5 0.0
nav_cancel
start demo_visual_inspection 1 0
pause <mission_id>
resume <mission_id>
cancel <mission_id>
estop on
estop off
report <mission_id>
recoveries
quit
```

`move LINEAR ANGULAR` 仅在 `mode manual` 下生效，默认硬限制为 ±0.4 m/s 和 ±1.2 rad/s（可用启动参数调整，但仍会受 `safety_mux` 限制）。`nav X Y YAW` 仅在 `mode nav` 下发送 Nav2 单点目标，且巡检任务运行时被拒绝。`estop off` 只会释放人工急停信号；任务仍必须再执行 `resume`，且传感器健康与人员安全条件必须满足。`recoveries` 列出 P5 中因 Mission Manager 重启而安全停在 `WAITING_OPERATOR` 的任务；根据返回的 `retry_current_checkpoint_index` 或 `continue_next_checkpoint_index` 使用 `start` 显式创建后续任务。

## 3. 自动化测试

在仓库根目录运行纯 Python 测试。这些测试不需要 ROS master 或硬件：

```bash
python3 -m unittest discover -s tests -v
```

在 ROS 2 工作空间内还应执行：

```bash
colcon test --packages-select \
  car_bringup car_control car_lidar car_map_manager car_mission \
  car_inspection car_navigation car_vision car_app_bridge
colcon test-result --verbose
```

### 一次性车端冒烟测试

已连接底盘、雷达和相机后，可运行以下脚本。它会依次检查控制/雷达、视觉、建图、导航、巡检和 APP 桥接，并将每个阶段的日志写到 `.run_logs/`：

```bash
cd ~/iCar-ROS2-SmartCar
bash scripts/run_full_system_smoke_test.sh
```

使用现场地图或检查 YOLO 时：

```bash
MAP=/data/maps/lab.yaml WITH_YOLO=true bash scripts/run_full_system_smoke_test.sh
```

脚本只验证节点、数据流、Action 与 Service，**不会**发布行驶速度、导航目标、巡检任务或解除急停命令。它可以发现软件、驱动、话题、相机、雷达、Nav2、任务接口和桥接接口的联调问题，但不能替代有人监护下的制动距离、实际绕障、二维码识别和视觉精度验收。

## 4. 按模块验收方法

| 模块 | 自动化检查 | 实车/ROS 检查 |
| --- | --- | --- |
| `car_bringup` | `tests/test_bringup_cli.py` | `icar start ... --dry-run`；验证 profile 参数和互斥启动规则 |
| `car_control` | `tests/test_safety_policy.py` | 发布人员急停、传感器故障、人工急停；确认 `/control/cmd_vel` 为零且人员清除延迟后才恢复 |
| `car_lidar` | 通过 `car_control` 安全契约测试 | 录制/回放 `/scan`，验证 `/lidar/override_active`、Nav2 局部避障和持续阻挡恢复 |
| `car_navigation` | `tests/test_navigation_safety_contract.py` | `ros2 action send_goal /navigate_to_pose ...`，确认 Nav2 只发布到 `/cmd_vel_nav` |
| `car_map_manager` | `tests/test_map_repository.py` | 导入 P2/P5 地图，验证可用栅格、占用格、未知格和边界点被正确判定 |
| `car_mission` | `tests/test_mission_core.py` | 依次测试 Nav2 成功/拒绝/失败、暂停恢复、定位超时、重启后 `recoveries`、阻挡上限 |
| `car_inspection` | `tests/test_checkin_core.py`、`tests/test_inspection_core.py` | 用 rosbag 验证二维码连续帧、相机断流、模型不匹配、`UNKNOWN` 不误报 |
| `car_vision` | 巡检核心测试覆盖结果契约 | RGB-D 人员距离从安全区进入停车区并离开，验证 `/vision/person_estop` 和延迟恢复 |
| `car_app_bridge` | `tests/test_app_protocol.py`、`tests/test_control_lease.py` | TCP 连接测试认证、租约超时停车、`mission_export` 和 `mission_recoveries` |
| P5 健康监测 | `tests/test_mission_core.py` | 暂停 `/scan`、RGB、`/odom` 任一 Topic，确认 `/system/sensor_fault`、停车、任务 `WAITING_OPERATOR` |

真实路线任务的联调入口为：

```bash
bash scripts/start_mission_test.sh
```

它会启动导航、巡检、视觉、健康监测与 APP 桥接，但不会自动让小车移动。设置初始位姿后，在第二个终端通过 `icar console` 显式启动任务。

## 5. 长时间测试

在封闭、低速、有人监护的场地，使用完整 mission profile 连续运行 30 分钟；首轮通过后再延长到 60 分钟。记录 `/mission/event`、`/system/health`、`/control/effective_estop` 和每次报告。任一急停、传感器故障或定位丢失后，必须确认小车停止且不会自行恢复运动。
