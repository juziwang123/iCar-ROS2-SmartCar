# iCar 智能小车 ROS2 项目需求分析报告

## 1. 文档概述

### 1.1 编写目的

本文档用于明确 iCar 智能小车 ROS2 项目的建设目标、业务场景、功能需求、非功能需求、约束条件与验收标准，作为后续设计、开发、测试和答辩材料编写的统一依据。

### 1.2 项目背景

本项目来源于北京交通大学软件工程学期实训 III，依托润开鸿 iCar 智能小车硬件平台，围绕“多传感器巡检与自主导航”展开。项目需结合 ROS2 通信机制、激光雷达、深度相机、环境传感器以及移动端控制能力，完成实车控制、环境感知、建图导航、目标识别与文档交付等工作。

### 1.3 参考资料

1. 工作区文档：项目汇总、需求分析报告（简略分析）、小车连接指南。
2. 仓库根文档 README。
3. ROS2 Humble 官方文档。
4. 小车厂家配套使用资料与课堂讲义。

## 2. 项目目标与范围

### 2.1 项目目标

项目目标是构建一个面向教学与实训场景的智能小车软件系统，使其具备以下能力：

1. 支持远程连接、命令下发和状态查看。
2. 支持底盘运动控制与 ROS2 基础通信。
3. 支持激光雷达环境感知、避障与跟随。
4. 支持 SLAM 建图与自主导航。
5. 支持基于视觉的目标检测与跟踪。
6. 支持项目代码、测试材料、使用手册等工程化交付物输出。

### 2.2 项目范围

本项目范围包含以下内容：

1. 本地开发环境搭建与代码仓库管理。
2. ROS2 节点划分、话题通信与启动管理。
3. 小车底盘控制接口封装。
4. 激光雷达、视觉模块和导航模块的功能集成。
5. 文档、测试、演示视频和答辩材料编制。

### 2.3 范围外内容

以下内容不作为本阶段必须完成项，但可作为后续扩展：

1. 多车协同编队。
2. 基于大模型的复杂语义任务交互。
3. 面向真实道路的长期稳定巡检部署。
4. 面向商业环境的云端运维系统。

## 3. 干系人与使用场景

### 3.1 干系人

| 角色 | 关注点 |
|------|--------|
| 指导教师 | 项目完整度、工程规范、答辩效果 |
| 项目组长 | 任务分配、进度控制、交付质量 |
| 项目组员 | 模块实现、联调效率、学习成果 |
| 实训设备管理员 | 硬件安全、设备规范使用 |
| 演示观众/验收人员 | 功能可见性、稳定性、易用性 |

### 3.2 典型使用场景

1. 开发人员在 WSL Ubuntu 22.04 中完成 ROS2 Humble 环境搭建、代码开发与仿真验证。
2. 组员通过热点、SSH 或 NoMachine 连接实车，启动底盘、雷达、相机等基础服务。
3. 操作者通过键盘或移动端 APP 控制小车移动，验证手动控制链路是否可用。
4. 小车在实验室环境中完成建图、保存地图并执行导航到目标点任务。
5. 小车在固定区域内完成避障、跟随、目标识别或巡检演示。

## 4. 现状分析

### 4.1 现有资源

1. 已具备 iCar 小车硬件平台，含 Jetson、激光雷达、深度相机和底盘控制板。
2. 已提供连接指南、课程安排、任务清单和基础使用资料。
3. 已建立 GitHub 仓库，具备按功能包划分的 ROS2 目录骨架。

### 4.2 当前问题

1. 仓库当前以目录骨架为主，核心功能代码尚未完整落地。
2. 实车环境与本地环境存在 ROS2 Foxy/Humble 差异，需要明确开发与部署边界。
3. 项目正式文档尚不完整，难以直接支撑中后期测试、验收和答辩。

## 5. 功能需求分析

### 5.1 功能模块划分

按照系统职责，项目划分为五个核心功能域：

1. 连接与基础运行管理。
2. 底盘控制与手动遥控。
3. 激光雷达感知。
4. 视觉识别与跟踪。
5. SLAM 建图与自主导航。

### 5.2 详细功能需求

#### FR-01 远程连接与设备访问

1. 系统应支持开发电脑与小车连接同一热点网络。
2. 系统应支持通过 SSH 登录小车。
3. 系统应支持通过 NoMachine 或 VNC 进行图形化远程访问。
4. 系统应支持查看小车 IP、ROS2 版本、节点列表和话题列表。

验收标准：开发人员能够在 10 分钟内完成联网、登录和远程桌面连接。

#### FR-02 ROS2 基础通信与启动管理

1. 系统应按功能包组织节点、参数、启动文件和配置文件。
2. 系统应支持基于 Launch 文件启动单模块或多模块功能。
3. 系统应支持通过 Topic 发布与订阅实现模块间通信。
4. 系统应为后续 Service 或 Action 扩展预留接口。

验收标准：至少完成一个统一启动入口，并能稳定拉起核心演示链路。

#### FR-03 底盘控制与手动遥控

1. 系统应支持前进、后退、左转、右转、停止等基本动作。
2. 系统应支持键盘遥控或 APP 遥控。
3. 系统应支持速度参数配置与紧急停止。
4. 系统应保证不同控制源之间具备基本优先级约束，避免冲突。

验收标准：操作者可通过控制命令稳定驱动小车完成基本位移。

#### FR-04 激光雷达避障

1. 系统应采集激光雷达扫描数据。
2. 系统应识别前方或周边障碍物距离。
3. 系统应在障碍物进入阈值范围时触发减速、停止或绕行策略。
4. 系统应支持阈值参数可配置。

验收标准：在实验室典型障碍物场景下，小车能在碰撞前完成停止或规避。

#### FR-05 激光雷达跟随与警卫

1. 系统应依据雷达数据识别前方目标相对距离变化。
2. 系统应实现保持期望距离的跟随行为。
3. 系统应支持区域入侵检测并输出告警信息。
4. 系统应支持启停控制与参数调节。

验收标准：在单目标场景下，系统能够完成基础跟随或警卫演示。

#### FR-06 视觉目标检测与追踪

1. 系统应支持接入 RGB 或 RGB-D 相机图像流。
2. 系统应支持颜色识别或 YOLO 目标检测。
3. 系统应根据图像目标位置输出追踪控制建议。
4. 系统应支持模型或阈值配置，以适配课堂场景。

验收标准：在固定光照条件下，系统可识别指定目标并给出可见的追踪响应。

#### FR-07 SLAM 建图

1. 系统应支持基于激光雷达完成场地二维地图构建。
2. 系统应支持地图保存与复用。
3. 系统应支持建图过程中的位姿更新与基本可视化。

验收标准：可生成一张可用于导航的室内地图文件。

#### FR-08 自主导航与巡航

1. 系统应支持加载已保存地图。
2. 系统应支持设置单个或多个导航目标点。
3. 系统应支持基于路径规划完成自动到点。
4. 系统应支持巡航过程中配合避障策略运行。

验收标准：小车能够在指定室内场景中完成至少一次导航到点或多点巡航演示。

#### FR-09 APP/上位机桥接

1. 系统应支持将外部控制命令转换为 ROS2 控制消息。
2. 系统应支持显示基础状态信息，至少包括运动控制相关状态。
3. 系统应支持后续接入图像或传感器信息展示的扩展。

验收标准：上位机或移动端可以与 ROS2 控制链路打通。

#### FR-10 文档与工程化交付

1. 系统应提供需求分析、开发报告、测试报告和使用手册。
2. 系统应提供可读的 README 和清晰的目录结构。
3. 系统应支持代码版本管理与阶段性成果留痕。

验收标准：答辩前可提交完整文档，答辩后可补充测试证据和最终说明。

### 5.3 进阶需求

1. 道面缺陷检测与风险区域识别。
2. 湿滑积水、危险异物等目标检测。
3. 基于语音或文字的大模型任务控制。
4. 多车协同与编队控制。
5. 基于视觉的自动循线。

进阶需求不影响基础验收，但可作为展示亮点和加分内容。

### 5.4 进阶需求实施原则

1. 进阶需求必须在主线功能稳定后分阶段接入。
2. 进阶需求必须复用已有控制、视觉、导航接口，避免另起一套控制链路。
3. 每项进阶需求都应有最小可行版本和单独验收标准。
4. 若时间不足，应优先保留能快速形成演示效果的功能，如自动循线、缺陷检测和积水异物检测。

## 6. 非功能需求分析

### 6.1 性能需求

1. 手动控制链路应具备可接受的响应速度，正常网络环境下控制延迟建议低于 500 ms。
2. 避障逻辑应具备实时性，能够在课堂安全速度范围内及时停车或转向。
3. 建图与导航过程应保持持续运行，不出现频繁崩溃或严重卡顿。

### 6.2 可靠性需求

1. 系统应支持连续运行至少 30 分钟用于演示。
2. 单个模块异常时，应便于单独重启，不影响其他模块定位问题。
3. 文档应对常见故障给出排查说明。

### 6.3 可维护性需求

1. 仓库应按功能包、文档、脚本、测试分层组织。
2. 关键参数、启动文件和配置文件应可独立调整。
3. 文档命名、章节结构与术语应保持统一。

### 6.4 可扩展性需求

1. 系统应支持后续新增传感器或算法模块。
2. 模块间应通过 ROS2 标准消息或清晰接口耦合，避免硬编码依赖。
3. 视觉与导航模块应允许替换不同算法实现。

### 6.5 安全性需求

1. 系统应支持急停机制。
2. 演示场景应优先限制在室内平整、可控区域。
3. 设备连接、上电和运动前应有检查步骤，避免误操作。

### 6.6 可用性需求

1. 初次接触项目的成员应能够根据使用手册完成环境准备和基础操作。
2. 常用命令和启动流程应清晰、可复制。
3. 使用手册应覆盖连接、运行、调试与故障处理。

## 7. 约束条件

1. 实训周期较短，只有约 10 天，需要先完成必做项，再评估进阶项。
2. 实车环境为 Jetson Ubuntu 20.04 + ROS2 Foxy，本地环境为 Ubuntu 22.04 + ROS2 Humble，存在版本差异。
3. 硬件资源需要共享，实车调试时段有限。
4. 网络环境可能不稳定，远程连接存在波动风险。

|------|----------|--------|------|

| FR-03 | 底盘控制与手动遥控 | 高 | 最基础演示能力 |
| FR-04 | 激光雷达避障 | 高 | 核心感知能力 |
| FR-07 | SLAM 建图 | 高 | 自主导航前提 |
| FR-08 | 自主导航与巡航 | 高 | 核心展示能力 |

| FR-05 | 跟随与警卫 | 中 | 与雷达避障同源，可迭代实现 |
| FR-09 | APP/上位机桥接 | 中 | 提升交互性 |
| FR-10 | 文档与工程化交付 | 高 | 明确列入提交物清单 |

| EF-05 | 多车协同 | 低 | 工程复杂度高，受设备条件限制 |

## 9. 风险分析与应对措施

| 风险 | 表现 | 应对措施 |
|------|------|----------|
| 硬件连接异常 | 无法识别雷达、相机或底盘 | 提前完成通电与接口检查，保留可回退配置 |
| ROS2 版本差异 | 本地能跑、实车报错 | 区分开发环境和部署环境，记录适配差异 |
| 算法调试时间不足 | 检测、跟随、导航效果不稳定 | 优先保证基础闭环，再做精细优化 |
| 网络不稳定 | SSH/远程桌面中断 | 准备热点备选方案，保留本地仿真验证路径 |
| 文档滞后 | 开发完成但材料不足 | 与开发同步维护文档，每日补充记录 |

## 10. 验收标准汇总

1. 能够完成实车连接、基础控制与状态查看。
2. 能够完成至少一个完整传感器驱动链路演示。
3. 能够完成建图与导航或视觉识别中的核心功能展示。
4. 仓库具备清晰目录结构与必要的说明文档。
5. 能提交需求分析报告、开发报告、测试报告、使用手册等正式材料。

## 11. 当前阶段结论

结合现有课程要求、硬件条件和仓库状态，本项目应采用“先文档建基线、再按优先级逐步实现模块”的推进方式。当前阶段最关键的需求不是追求一次性覆盖全部进阶能力，而是建立稳定的连接、控制、感知与导航主链路，并形成可验收、可复现、可答辩的工程文档体系。
