# iCar 智能小车 ROS2 项目需求分析报告

## 1. 文档概述

### 1.1 编写目的

本文档基于现有课程材料，对 iCar 智能小车 ROS2 项目的建设目标、课程驱动需求、实现范围、约束条件与验收标准进行重构。文档用于统一项目组对“课程要求是什么、项目必须做什么、阶段成果如何验收”的理解，并为后续设计文档、开发报告、测试报告和答辩材料提供依据。

### 1.2 项目背景

本项目来源于北京交通大学软件工程学期实训 III，依托润开鸿 iCar 智能小车平台展开。课程材料已经覆盖底盘固件烧录、环境感知开发、激光雷达建图导航、激光雷达跟踪避障、计算机视觉追踪以及鸿蒙 APP 开发六个教学模块，因此项目需求不应再仅按抽象的软件模块描述，而应直接围绕课程提供的能力链路进行设计。

### 1.3 重构依据

本次文档重构主要依据以下材料：

1. 课程材料目录“智能小车课程材料”六大模块。
2. 底盘固件包中的说明文件，明确 `iCar.bin` 用于 ROS 常规功能，`follow-line.bin` 用于循线与 IoT 演示。
3. 项目汇总文档中列出的必做任务、加分项、交付物和考核方式。
4. 小车连接指南中给出的联网、SSH、NoMachine、VNC 与常用 ROS2 运维流程。
5. 仓库现有工程骨架与已有正式文档体系。

## 2. 课程材料分析

### 2.1 课程模块拆解

| 课程模块 | 主要内容 | 对项目需求的直接影响 |
|------|------|------|
| 1. 智能小车使用和固件烧录 | 设备使用、底盘固件切换、烧录工具 | 项目必须明确不同演示场景对应的固件模式与切换要求 |
| 2. 环境感知开发 | 温湿度、光照、GPS、气压、PM2.5、可燃气体、WiFi、NB、姿态、电机、编码器、PID | 项目必须具备多传感器接入与底盘控制基础，不能只关注雷达与视觉 |
| 3. 激光雷达 SLAM 建图及自动导航 | 仿真建图导航、实车建图导航 | 项目必须同时考虑仿真验证链路和实车落地链路 |
| 4. 基于激光雷达跟踪及避障 | A1 雷达使用、避障、跟随、警卫 | 项目必须将避障、跟随、警卫视为独立可验收能力 |
| 5. 基于计算机视觉的目标追踪 | 深度相机使用、视觉追踪、目标检测 | 项目必须包含图像接入、目标识别与闭环追踪需求 |
| 6. 鸿蒙 APP 开发 | 环境搭建、UI、摇杆交互、TCP 通信 | 项目必须把移动端控制链路纳入主线需求，而不是可选附属功能 |

### 2.2 课程驱动的需求重心

结合课程安排与材料内容，本项目的需求重心应从“泛化的智能车平台”收敛为“可在实训周期内完成演示和答辩的教学型智能车系统”，同时明确采用“基础功能保底、高阶功能拉开差异”的双层建设思路，即：

1. 优先完成课程明确覆盖的能力闭环。
2. 优先完成可以被课堂验收直接观察到的功能。
3. 优先复用课程现成硬件、固件、驱动、实验流程与操作方式。
4. 在主线稳定基础上，必须形成至少 1 至 2 个可演示的高阶功能，而不是只停留在课程基础能力复现。

## 3. 项目目标与范围

### 3.1 项目目标

项目目标是构建一套与课程材料一致、可在教学与答辩场景中稳定运行的智能小车软件系统，使其具备以下能力：

1. 支持设备连接、远程访问、底盘固件匹配和基础运维。
2. 支持底盘运动控制、编码器/PID 调节和基础传感器接入。
3. 支持激光雷达建图、保存地图、自主导航和局部避障。
4. 支持基于激光雷达的跟随、警卫等课程预置能力。
5. 支持基于相机的目标检测、目标追踪或自动循线等视觉能力。
6. 支持 APP 或上位机通过网络协议接入控制链路。
7. 支持多传感器融合、视觉风险识别、任务级智能交互或多车协同等高阶能力中的若干项。
8. 支持按课程要求输出完整文档、代码仓库、测试证据与演示材料。

### 3.2 项目范围

本项目范围包括：

1. 基于 ROS2 的功能包组织、节点通信与启动管理。
2. 小车实车连接、远程桌面访问与常用运维命令整理。
3. 底盘控制、传感器数据接入、感知与控制闭环集成。
4. 雷达、视觉、导航和 APP 控制链路实现或打通。
5. 高阶功能原型设计、联调验证与演示方案建设。
6. 仿真、实车、测试、文档与答辩交付物建设。

### 3.3 范围外内容

以下内容不作为本阶段硬性完成项：

1. 面向真实复杂道路的长期自主巡检。
2. 商业化、生产级云端运维平台建设。
3. 完整开放域自然语言智能体控制。
4. 与课程主线无直接关系的大规模算法创新。

## 4. 干系人与场景

### 4.1 干系人

| 角色 | 关注点 |
|------|--------|
| 指导教师 | 是否覆盖课程主线、是否满足答辩与验收要求 |
| 项目组长 | 进度、风险、成果整合与提交质量 |
| 模块开发成员 | 自己负责模块的可实现性、联调边界与复用效率 |
| 设备管理员 | 固件、硬件、安全与设备完好性 |
| 验收人员/答辩评委 | 功能完整度、演示效果、工程规范与文档质量 |

### 4.2 典型使用场景

1. 组员在本地 WSL 或 Ubuntu 环境完成 ROS2 开发，并在仿真环境先验证建图导航流程。
2. 开发人员通过手机热点让电脑与小车接入同一网络，再通过 SSH 或 NoMachine 连接实车。
3. 小车根据当前实验内容切换底盘固件，例如普通 ROS 功能使用 `iCar.bin`，循线演示使用 `follow-line.bin`。
4. 操作者使用键盘、APP 或 TCP 控制命令驱动小车运动并查看状态。
5. 小车在实验室中完成建图、导航、避障、跟随、警卫、视觉检测或目标追踪演示。

## 5. 现状分析

### 5.1 已具备条件

1. 课程材料已经完整给出六个教学模块，项目需求依据更明确。
2. 已具备 iCar 小车、Jetson、思岚 A1 激光雷达、Astra 深度相机及多种环境传感器。
3. 已有项目汇总、连接指南、需求分析简版、README 和正式文档目录。
4. 仓库已经具备按功能包划分的 ROS2 工程骨架。

### 5.2 当前问题

1. 现有需求分析偏“通用软件项目”表述，没有把课程材料拆成可执行需求。
2. 文档中对固件切换、传感器实验链路、APP TCP 通信链路体现不足。
3. 仓库实现仍以工程骨架为主，正式文档需要先把主线边界描述清楚，避免开发目标发散。
4. 当前报告对高阶功能的定位偏弱，尚未体现“课程基础能力之上还要做项目亮点”的要求。

## 6. 功能需求分析

### 6.1 课程主线功能域

按照课程材料与项目亮点要求，项目功能域重新划分为八类：

1. 设备接入与固件管理。
2. 底盘控制与环境感知基础能力。
3. 激光雷达建图与自主导航。
4. 激光雷达避障、跟随与警卫。
5. 视觉检测、追踪与循线扩展。
6. APP/上位机交互控制。
7. 高阶智能与创新展示能力。
8. 文档、测试与交付管理。

### 6.2 详细功能需求

#### FR-01 设备连接与远程访问

1. 系统应支持电脑与小车连接同一热点网络。
2. 系统应支持查看小车 IP 并验证网络连通性。
3. 系统应支持通过 SSH 登录小车。
4. 系统应支持通过 NoMachine 或 VNC 远程图形化访问。
5. 系统应支持查看 ROS2 版本、节点列表与话题列表。

验收标准：新成员可按照连接指南在 10 分钟内完成联网、登录和基础 ROS2 状态检查。

#### FR-02 底盘固件管理

1. 系统应明确不同实验场景所需的底盘固件。
2. 系统应支持普通 ROS 功能场景使用 `iCar.bin`。
3. 系统应支持循线或 IoT 采集演示场景使用 `follow-line.bin`。
4. 系统应保留底盘固件烧录工具与切换说明，避免因固件不匹配导致功能异常。

验收标准：项目文档能够说明至少两类固件的适用场景，组员可按说明完成切换。

#### FR-03 底盘控制与运动调节

1. 系统应支持前进、后退、左转、右转、停止等基本底盘动作。
2. 系统应支持速度参数配置、限速和紧急停止。
3. 系统应支持编码器、PID 等底盘运动调节能力的接入或参数配置。
4. 系统应支持多控制源接入时的冲突规避，例如键盘、APP 和自治模块之间的优先级划分。

验收标准：操作者能够稳定驱动小车完成基本位移，并且参数调整后小车运动行为有可见变化。

#### FR-04 环境感知与基础传感器接入

1. 系统应支持接入课程材料涉及的环境感知类传感器数据。
2. 传感器接入范围至少应覆盖温湿度、光照、气压、PM2.5、可燃气体、GPS、姿态中的若干项。
3. 系统应支持统一采集、发布或显示传感器数据，便于后续可视化和扩展。
4. 系统应支持网络通信模块接入，为 WiFi/NB 或上云实验预留边界。

验收标准：至少完成一条环境传感器采集链路，并能在终端、日志或界面中查看结果。

#### FR-05 ROS2 基础通信与启动管理

1. 系统应按功能包组织节点、配置文件、启动文件和资源目录。
2. 系统应支持通过 Launch 文件启动底盘、雷达、视觉、导航等模块。
3. 系统应支持基于 Topic 的消息发布与订阅。
4. 系统应为 Service、Action 或 TCP 桥接保留扩展接口。

验收标准：至少形成一套可以稳定拉起核心演示链路的启动方案。

#### FR-06 激光雷达 SLAM 建图

1. 系统应支持思岚 A1 激光雷达接入与扫描数据获取。
2. 系统应支持先在仿真环境验证建图流程，再在实车环境复现。
3. 系统应支持生成二维栅格地图并保存结果。
4. 系统应支持建图过程中位姿更新与可视化观察。

验收标准：能够在课程实验场地或仿真场景中生成一张可复用地图文件。

#### FR-07 自主导航

1. 系统应支持加载已保存地图。
2. 系统应支持设置单点或多点导航目标。
3. 系统应支持根据规划路径控制底盘自动到点。
4. 系统应支持导航过程中与局部避障策略协同运行。

验收标准：小车能够在指定室内场景完成至少一次导航到点演示。

#### FR-08 激光雷达避障

1. 系统应支持根据激光雷达数据识别障碍物距离。
2. 系统应支持阈值配置，并在障碍物接近时触发减速、停车或绕行。
3. 系统应支持在课程演示速度范围内稳定运行。

验收标准：在典型障碍物场景下，小车能够在发生碰撞前停止或规避。

#### FR-09 激光雷达跟随与警卫

1. 系统应支持根据雷达数据识别目标距离变化。
2. 系统应支持保持设定距离的跟随行为。
3. 系统应支持固定区域的警卫或入侵检测逻辑。
4. 系统应支持启停控制与参数调节。

验收标准：在单目标教学场景中，能够完成一次稳定跟随或警卫演示。

#### FR-10 视觉目标检测与追踪

1. 系统应支持 Astra 深度相机或其他相机图像流接入。
2. 系统应支持基于颜色、传统视觉或 YOLO 的目标检测。
3. 系统应支持依据目标位置输出追踪控制信号。
4. 系统应支持在固定光照与课程指定目标下完成演示。

验收标准：小车或系统界面能够识别指定目标，并产生可观测的追踪响应。

#### FR-11 视觉循线与扩展视觉任务

1. 系统应支持将自动循线作为与课程固件相匹配的扩展演示项。
2. 系统应支持在视觉链路稳定后扩展至道面缺陷检测、积水检测或危险异物检测。
3. 系统应优先复用现有相机接入、检测框架与底盘控制接口。

验收标准：至少明确一项视觉扩展任务的实现路径或最小可行版本。

#### FR-12 APP/上位机控制链路

1. 系统应支持移动端或上位机通过网络协议向小车发送控制命令。
2. 系统应体现鸿蒙 APP 实验手册涉及的 UI、摇杆交互和 TCP 通信链路。
3. 系统应支持将外部命令转换为 ROS2 控制消息或底盘控制指令。
4. 系统应支持显示基础状态信息，并为传感器或图像展示预留扩展接口。

验收标准：APP 或上位机可成功打通至少一条远程控制链路。

#### FR-13 文档与工程化交付

1. 系统应提供需求分析报告、设计文档、项目开发报告、测试报告和使用手册。
2. 系统应维护 README、目录结构说明、阶段性日报和提交清单。
3. 系统应保留代码仓库、演示视频和阶段性证据。
4. 系统应支持中期检查与最终答辩两阶段展示需求。

验收标准：在课程截止前能够提交完整文档与代码材料，并支撑答辩说明。

### 6.3 高阶功能需求

高阶功能不是对课程基础功能的简单重复，而是基于已有底盘、雷达、视觉、导航和控制链路做进一步集成、智能化和展示深化。项目至少应落地 1 至 2 项高阶能力，并形成独立演示闭环。

#### HF-01 道面缺陷检测与避障

1. 系统应支持识别裂缝、坑洞或破损区域等道面缺陷目标。
2. 系统应支持在识别出缺陷后触发减速、停车或绕行策略。
3. 系统应优先复用现有视觉检测框架与控制接口。

验收标准：在预设场景中能稳定识别至少一种缺陷目标，并触发对应安全动作。

#### HF-02 湿滑积水与危险异物检测

1. 系统应支持识别积水、反光湿滑区域或危险异物。
2. 系统应支持对风险目标输出告警、标框或控制动作。
3. 系统应支持将检测结果与导航或避障策略结合，形成风险规避闭环。

验收标准：至少对一种风险目标给出检测结果，并产生可见系统响应。

#### HF-03 多传感器融合感知

1. 系统应支持将视觉、激光雷达和部分环境传感器信息进行统一处理或联合决策。
2. 系统应支持将融合结果用于风险判断、状态显示或任务执行。
3. 系统应支持在界面、日志或控制决策中体现融合后的信息价值。

验收标准：至少完成一条“多源感知输入-融合判断-动作或展示输出”的闭环演示。

#### HF-04 LLM 或任务级智能交互

1. 系统应支持将自然语言或模板任务指令转换为小车可执行动作。
2. 系统应支持至少 3 至 5 条高层任务模板，例如巡航、警卫、检测、跟随或停止。
3. 系统应限制高阶任务调用边界，避免绕过底层安全控制链路。

验收标准：至少完成一次“输入任务-解析任务-执行动作-反馈状态”的闭环演示。

#### HF-05 多车协同或队列控制

1. 在设备条件允许时，系统应支持两台及以上小车的基础协同行为。
2. 系统应支持主从控制、同步启动停止、简化跟车或共享目标点中的至少一种模式。
3. 若实车条件不足，应提供命名空间仿真、单车模拟或设计原型作为替代方案。

验收标准：至少完成一次协同动作演示，或提交可运行原型与可验证设计说明。

#### HF-06 云平台集成与 CI/CD

1. 系统应支持代码仓库的规范管理，并接入 GitHub、Gitee 或课程要求的平台。
2. 系统应支持基础的自动化检查、构建验证或成果同步流程。
3. 系统应支持保留构建记录、测试记录或平台接入截图，作为工程化证据。

验收标准：至少形成一条可复现的代码提交到自动检查或平台同步链路。

### 6.4 高阶功能实施原则

1. 高阶功能必须建立在课程主线功能稳定的前提下，但不能完全后置到项目最后一天才开始。
2. 高阶功能应优先选择复用度高、演示效果强、实现边界清晰的方向。
3. 每个高阶功能都应具备最小可行版本、负责人、依赖项与独立验收标准。
4. 若时间不足，应优先保留能形成差异化答辩亮点的功能，而不是平均分散投入。
5. 高阶功能不得绕过急停、避障和底盘安全控制逻辑直接驱动车辆。

## 7. 非功能需求

### 7.1 性能需求

1. 手动控制链路在正常网络环境下应具备可接受响应速度，建议控制延迟低于 500 ms。
2. 避障逻辑应在课堂安全速度下具备实时响应能力。
3. 建图、导航、视觉识别演示过程中不应频繁崩溃或长时间卡顿。

### 7.2 可靠性需求

1. 系统应支持连续运行至少 30 分钟用于演示或答辩。
2. 单模块异常时应支持单独重启和快速定位问题。
3. 仿真链路应作为实车异常时的备选演示路径。

### 7.3 可维护性需求

1. 仓库应按功能包、文档、脚本和测试分层组织。
2. 启动文件、阈值参数、模型路径和地图文件应可独立维护。
3. 文档术语应统一使用课程中的设备名称和功能名称，避免前后不一致。

### 7.4 可扩展性需求

1. 系统应支持新增传感器、算法模型或控制前端。
2. 模块间应优先通过 ROS2 标准消息或清晰桥接接口解耦。
3. 视觉与导航模块应允许后续替换不同算法实现。
4. 系统架构应能容纳高阶功能节点接入，而不破坏基础主链路稳定性。

### 7.5 安全性需求

1. 系统应具备急停机制。
2. 演示场地应限定在室内平整、可控区域。
3. 每次运行前应执行网络、供电、固件和传感器状态检查。
4. 固件切换后应重新验证底盘控制行为，防止误用错误固件。

### 7.6 可用性需求

1. 新成员应能根据连接指南和使用手册完成基础环境准备。
2. 常用连接、启动、排障命令应清晰可查。
3. APP 或上位机的控制方式应尽量保持直观，便于课堂演示。

## 8. 约束条件

1. 实训周期短，功能实现与文档产出需要并行推进。
2. 实车环境、课程环境与本地开发环境之间可能存在 ROS 版本差异。
3. 硬件设备共享，实车调试时段有限。
4. 远程连接依赖热点或局域网，网络波动会影响开发效率。
5. 部分演示依赖固件模式切换，不同功能不能保证在同一固件下全部完成。
6. 课程验收强调可见演示效果，因此需求实现必须兼顾稳定性与展示性。
7. 高阶功能开发必须受时间、算力、设备数量和数据集准备情况约束。

## 10. 风险分析与应对措施

| 风险 | 表现 | 应对措施 |
|------|------|----------|
| 固件不匹配 | 底盘行为异常或功能无法运行 | 明确每项演示对应固件，切换后先做最小动作验证 |
| ROS2 版本差异 | 本地可运行，实车部署报错 | 区分开发环境与部署环境，记录版本适配策略 |
| 传感器联调时间不足 | 雷达、相机、环境传感器无法同时稳定工作 | 先完成主链路，再逐项叠加传感器 |
| 网络不稳定 | SSH、NoMachine 或 APP 控制中断 | 准备热点备选方案，保留仿真和离线演示路径 |
| 算法调参成本高 | 检测、导航或跟随效果不稳定 | 优先保证能演示，再优化精度和速度 |
| 高阶功能范围过大 | 多个方向同时推进导致全部做浅 | 设立 1 至 2 个主攻高阶方向，其余保留原型或设计说明 |
| 数据和设备不足 | 缺陷检测、多车协同难以稳定复现 | 使用固定场景、预设样本、仿真或录屏证据降低风险 |
| 文档滞后 | 实现完成但无法支撑答辩 | 开发与文档同步推进，按模块沉淀证据 |

## 11. 验收标准汇总

1. 能够完成小车联网、远程登录和基础 ROS2 运维检查。
2. 能够说明并正确切换课程所需底盘固件。
3. 能够完成底盘基本运动控制，并具备最小安全控制能力。
4. 能够完成至少一条环境感知或基础传感器采集链路。
5. 能够完成建图与导航主链路，至少达到一次可复现演示。
6. 能够完成避障、跟随、警卫、视觉检测、APP 控制中的若干重点演示项。
7. 能够完成至少 1 至 2 个高阶功能闭环演示，或提供可验证原型与设计说明。
8. 能够提交与课程要求一致的正式文档、代码和演示材料。

## 12. 当前阶段结论

在课程材料已经完整具备的前提下，本项目的需求分析不应继续沿用抽象的软件项目模板，而应显式围绕“课程模块、实验链路、设备能力、演示要求、交付清单、高阶亮点”六条主线展开。重构后的需求报告将项目主目标收敛为：先打通连接、控制、建图导航、避障跟随、视觉追踪和 APP 控制等课程基础链路，再至少完成若干项高阶功能闭环，以形成既满足课程要求、又具备项目辨识度和答辩竞争力的智能小车系统。
