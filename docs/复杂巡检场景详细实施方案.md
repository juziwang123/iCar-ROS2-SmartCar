# iCar 复杂巡检场景详细实施方案

## 1. 文档目的

本文档面向 iCar ROS2 智能小车在真实室内或半室外固定场景中的自主巡检需求，给出从首次建图、APP 路线规划、检查点打卡、视觉巡检、异常处理到人工接管的完整实现方案。

本文档不是仅描述最终效果，而是明确：

1. 当前仓库中哪些能力可以复用。
2. 当前实现中必须优先解决的安全和集成问题。
3. 应新增哪些 ROS2 包、节点、消息、Action 和 Service。
4. 路线、检查点、巡检任务和证据应如何建模与存储。
5. APP 应提供哪些页面、指令和状态反馈。
6. 各阶段的开发顺序、代码改造范围和验收指标。

目标场景如下：小车在指定场地内按照 APP 配置的路线巡逻，在指定检查点完成可靠打卡，并拍照检查门前是否存在消防设施等目标；遇到障碍物、人员阻挡、定位失败或识别失败时能够安全降级；操作者可随时通过 APP 暂停并接管小车，同时查看实时路线和任务完成情况。

## 2. 建设目标与范围

### 2.1 最终目标

系统应形成以下完整闭环：

```text
场地建图
  -> 地图保存与版本管理
  -> APP 在地图上规划路线和检查点
  -> 路线合法性校验
  -> 小车自主定位并逐点导航
  -> 检查点到达确认和视觉打卡
  -> 拍照执行巡检任务
  -> 本地模型或视觉大模型给出结构化结论
  -> 异常恢复或人工处理
  -> APP 实时显示进度
  -> 生成包含图片和事件的巡检报告
```

### 2.2 第一阶段不建议纳入的范围

以下内容会显著增加复杂度，不应阻塞单车巡检主线：

1. 多车协同调度。
2. 完全无人值守的自动充电。
3. 在未知环境中自主探索并自动决定巡检路线。
4. 让 LLM 直接生成底盘速度或绕过安全仲裁。
5. 在控制 TCP 通道中传输实时视频或大尺寸图片。

### 2.3 推荐的最小可行产品（MVP）

第一版应完成：

1. 加载已经保存的二维地图。
2. APP 在地图上添加、排序和保存检查点。
3. 小车通过 Nav2 按顺序导航。
4. 使用“定位围栏 + 二维码”完成检查点打卡。
5. 到点自动拍照，使用 YOLO 或 VLM 判断目标是否存在。
6. 障碍物出现时停车、等待并尝试重新规划。
7. APP 可暂停、接管、恢复或取消任务。
8. 保存每个检查点的时间、位置、图片、识别结果和异常事件。

## 3. 当前代码能力评估

### 3.1 可复用能力

| 现有模块 | 当前能力 | 在新系统中的定位 |
| --- | --- | --- |
| `car_navigation/launch/mapping.launch.py` | 启动 SLAM Toolbox、可选 Scan Filter 和静态 TF | 作为建图模式基础入口 |
| `car_navigation/launch/navigation.launch.py` | 启动 Nav2、地图服务器和 AMCL | 作为定位和导航基础入口 |
| `car_navigation/config/nav2_params.yaml` | DWB、全局/局部 Costmap、雷达障碍层、恢复行为 | 继续负责路径规划和常规动态避障 |
| `car_navigation/waypoint_patrol.py` | 从 YAML 读取 `(x, y, yaw)` 并逐点发送 `NavigateToPose` | 仅作为原型参考，后续由任务编排节点替换 |
| `car_control/safety_mux.py` | 手动、导航、视觉、跟随、雷达速度源仲裁 | 改造成唯一的运动安全入口 |
| `car_control/motion_controller.py` | 速度限幅并发布到底盘 `/cmd_vel` | 保留为最终底盘输出节点 |
| `car_lidar` | 雷达避障、告警、目标跟随原型 | 常规避障由 Nav2 负责；近距离安全检测可复用 |
| `car_vision/yolo_detector.py` | YOLO 跟踪、RGB-D 人员距离、人员减速/停车信号 | 拆分人员安全检测与巡检目标检测职责 |
| `car_app_bridge` | TCP JSON Lines 指令、认证、单点导航、轻量遥测 | 升级为 v2 协议，接入路线和任务接口 |
| `tests/test_app_protocol.py` | APP 协议纯函数测试 | 扩展为路线、任务和状态机测试 |

### 3.2 现有航点巡逻的限制

当前 `waypoint_patrol.py` 只能：

1. 加载简单坐标数组。
2. 向 Nav2 依次发送目标。
3. 收到 Action 结果后切换到下一点。

它暂不具备：

1. 检查点 ID、名称和类型。
2. 到点停留、二维码打卡或拍照任务。
3. Action 成功、失败、取消状态的正确分类。
4. 失败重试、跳过、暂停、恢复和任务取消。
5. APP 动态路线下发。
6. 进度反馈和持久化。
7. 程序重启后的安全恢复。

因此不建议继续在该文件中增加大量 `if/else`。应新增独立的任务编排包，由任务编排器调用 Nav2。

### 3.3 当前必须先修复的 P0 问题

#### P0-1：Nav2 速度可能绕过安全仲裁

当前控制链设计是：

```text
/cmd_vel_manual / /cmd_vel_nav / /cmd_vel_vision / ...
        -> safety_mux
        -> /control/cmd_vel
        -> motion_controller
        -> /cmd_vel
        -> 底盘
```

但 Nav2 默认发布 `/cmd_vel`，而 `safety_mux` 订阅 `/cmd_vel_nav`。当前导航启动文件没有对 Nav2 的 `/cmd_vel` 做 remap，可能造成 Nav2 直接控制底盘并绕过急停、人员安全和模式仲裁。

实施要求：

1. 在 Nav2 的 Launch 作用域中将 `/cmd_vel` remap 为 `/cmd_vel_nav`。
2. 系统中只有 `motion_controller` 可以发布最终 `/cmd_vel`。
3. 增加集成测试，启动完整系统后检查 `/cmd_vel` 发布者数量。
4. 如果发现厂商底盘节点也订阅其他速度话题，应统一到同一出口。

#### P0-2：人员急停信号没有真正截断输出

当前 `safety_mux` 订阅了 `/vision/person_estop`，但最终发布逻辑只检查普通 `estop_active`。应将停车条件修改为：

```text
effective_estop = operator_estop
               OR person_estop
               OR sensor_fault_estop
               OR mission_safety_stop
```

只要 `effective_estop` 为真，立即发布零速度，且不能继续选择其他速度源。

#### P0-3：急停应锁存

普通障碍消失后可以自动恢复，但人工急停不应自动解除。建议区分：

| 信号 | 是否锁存 | 恢复方式 |
| --- | --- | --- |
| APP/物理急停 | 是 | 操作者显式确认释放 |
| 人员近距离停车 | 否 | 人员离开并连续安全一段时间 |
| 雷达近距离停车 | 否 | 障碍移开并满足迟滞阈值 |
| 传感器故障停车 | 是或半锁存 | 传感器恢复并通过健康检查 |

#### P0-4：多模式启动必须互斥

建图、导航和巡检不应同时启动。`bringup.launch.py` 当前允许多个布尔开关同时为真，后续应增加统一 `system_mode`：

```text
mapping | localization | patrol | manual_test
```

Launch 或 Mode Manager 应拒绝冲突组合，避免同时启动两个 Nav2 实例或同时运行建图和定位。

#### P0-5：实车验证尚未完成

地图、TF、导航和视觉模型的存在不代表实车已经闭环。进入任务开发前至少验证：

1. `/scan` 方向、角度和距离正确。
2. `map -> odom -> base_footprint -> laser` TF 连续且无重复发布者。
3. AMCL 初始位姿设置后能够稳定定位。
4. 单点导航能够正确到达并返回 Nav2 成功状态。
5. 导航速度确实经过 `safety_mux`。
6. 急停、断连和近距离人员停车均能截断底盘输出。

#### 4.2.4 `car_map_manager`

可单独创建包，也可在初期并入 `car_mission`。长期建议独立，职责为：

1. 调用 Nav2 `SaveMap` Service 保存地图。
2. 读取地图 YAML 和图像。
3. 生成地图清单和缩略图。
4. 管理 `map_id`、版本和校验和。
5. 验证路线绑定的地图版本。

#### 4.2.5 `car_health`

后期增加健康监测节点，订阅传感器和关键状态，输出统一 `SystemHealth`：

1. 雷达数据是否超时。
2. RGB/Depth 图像是否超时。
3. TF 是否可查询。
4. AMCL 定位质量。
5. Nav2 生命周期状态。
6. 底盘里程计是否更新。
7. 电量和温度状态。
8. 网络和 VLM 服务可用性。

## 5. 地图与坐标系统实现

### 5.1 首次建图流程

APP 发起建图任务后执行：

1. Mode Manager 确认当前没有导航或巡检任务。
2. 启动底盘、雷达、里程计、TF 和 SLAM Toolbox。
3. `safety_mux` 切换到 `manual`。
4. APP 通过摇杆持续发送手动速度和心跳。
5. APP 订阅低频地图预览与小车位姿。
6. 操作者沿场地边缘、走廊和关键区域缓慢移动。
7. 尽量形成闭环，等待 SLAM 回环优化稳定。
8. APP 输入地图名称并调用受控 `SaveMap` Service。
9. 地图管理器校验名称，调用 Nav2 Map Saver。
10. 保存地图元信息和校验和。
11. 停止 SLAM，进入 localization 模式验证地图。

### 5.2 地图目录设计

运行时资源不要写入 ROS 安装目录，建议使用：

```text
~/.icar/
├── maps/
│   └── <map_id>/
│       ├── map.yaml
│       ├── map.pgm
│       ├── preview.png
│       └── manifest.json
├── evidence/
├── reports/
├── logs/
└── icar.db
```

`manifest.json` 示例：

```json
{
  "map_id": "map_lab_20260711_001",
  "name": "实验室一层",
  "created_at": "2026-07-11T15:30:00+08:00",
  "resolution": 0.05,
  "origin": [-6.0, -4.03, 0.0],
  "image": "map.pgm",
  "sha256": "...",
  "frame_id": "map"
}
```

### 5.3 APP 地图坐标转换

地图 YAML 包含：

```text
resolution: 每个像素代表的米数
origin: 地图图像左下角在 map 坐标系中的 x、y、yaw
```

当 `origin_yaw = 0` 时，地图坐标转图片像素可使用：

```text
u = (x - origin_x) / resolution
v = image_height - 1 - (y - origin_y) / resolution
```

APP 点击像素转地图坐标：

```text
x = origin_x + u * resolution
y = origin_y + (image_height - 1 - v) * resolution
```

若地图 origin 含旋转角，需要先做二维旋转矩阵变换，不能继续使用简化公式。

APP 与小车之间始终传输 `map` 坐标，不应传输屏幕像素坐标。

### 5.4 初始位姿

加载地图后必须完成 AMCL 初始位姿设置。APP 应支持：

1. 在地图上选择小车大概位置。
2. 拖动箭头设置朝向。
3. 发送 `PoseWithCovarianceStamped` 到 `/initialpose`，或调用封装 Service。
4. 显示 AMCL 当前位姿和协方差。
5. 定位质量达标前禁止启动巡检。

定位质量判断不应只看是否收到了 `/amcl_pose`，还应检查：

1. 消息新鲜度。
2. x/y 协方差。
3. yaw 协方差。
4. `map -> base_footprint` TF 是否稳定。
5. 短时间内是否出现大幅跳变。

## 6. 路线与检查点数据模型

### 6.1 路线配置示例

```yaml
schema_version: 1
route_id: route_lab_fire_01
map_id: map_lab_20260711_001
name: 实验室消防巡检路线
version: 3
enabled: true
loop: false
default_speed_profile: indoor_slow
created_at: "2026-07-11T16:00:00+08:00"

checkpoints:
  - checkpoint_id: CP-01
    sequence: 1
    name: 东门消防点
    type: inspection
    pose:
      frame_id: map
      x: 1.20
      y: -0.40
      yaw: 1.57
    arrival:
      position_tolerance_m: 0.30
      yaw_tolerance_rad: 0.35
      dwell_sec: 2.0
      max_pose_covariance: 0.25
    checkin:
      method: visual_marker
      marker_type: qr
      expected_marker_id: ICAR:CP-01
      timeout_sec: 8.0
      retries: 2
    tasks:
      - task_id: CP-01-T01
        type: visual_presence
        target: fire_extinguisher
        required: true
        capture_count: 3
        local_model: fire_safety_yolo_v1
        use_vlm_fallback: true
        confidence_threshold: 0.70
    failure_policy:
      navigation: retry_then_wait_operator
      checkin: alert_and_continue
      required_task: wait_operator

  - checkpoint_id: CP-02
    sequence: 2
    name: 西侧途经点
    type: transit
    pose:
      frame_id: map
      x: 3.00
      y: 0.50
      yaw: 0.0
    arrival:
      position_tolerance_m: 0.40
      yaw_tolerance_rad: 0.50
      dwell_sec: 0.0
    checkin:
      method: geofence
    tasks: []
```

### 6.2 检查点类型

| 类型 | 说明 | 到点后行为 |
| --- | --- | --- |
| `transit` | 仅用于约束路线走向 | 到达后立即进入下一点 |
| `checkin` | 只要求打卡 | 定位或视觉打卡后继续 |
| `inspection` | 需要执行检查任务 | 打卡、拍照、识别、记录 |
| `wait` | 固定等待点 | 等待指定时长或外部确认 |
| `operator` | 必须人工确认 | APP 确认后继续 |

### 6.3 检查点失败策略

每个检查点应显式配置失败策略，不能把所有失败都写死为同一种行为：

| 策略 | 适用场景 | 行为 |
| --- | --- | --- |
| `retry_then_continue` | 非关键打卡 | 重试后记录失败并继续 |
| `retry_then_skip` | 可跳过途经点 | 重试后跳到下一点 |
| `retry_then_wait_operator` | 关键巡检点 | 停车并等待 APP 指令 |
| `abort_mission` | 安全或强制任务失败 | 终止任务并保持停车 |
| `return_home` | 电量低或任务取消 | 导航至预设安全点；无法返航则停车 |

### 6.4 路线存储

推荐组合：

1. SQLite 保存路线索引、版本、任务实例、检查结果和事件。
2. YAML/JSON 保存可导出、可审查的路线定义。
3. 文件系统保存图片和报告。

建议数据库表：

```text
maps
routes
route_checkpoints
checkpoint_tasks
missions
mission_checkpoints
inspection_results
evidence_files
mission_events
control_leases
```

任务运行时使用路线快照，避免操作者修改路线后影响正在执行的任务。`missions` 表中应记录启动时的 `route_id + route_version`。

### 6.5 路线合法性校验

APP 保存路线前，小车端执行以下校验：

1. `map_id` 存在且校验和一致。
2. 检查点 ID 唯一，sequence 连续。
3. 坐标、角度、容差均为有限数值。
4. 航点落在空闲栅格内。
5. 航点距离占据栅格和未知区域满足安全距离。
6. 相邻点之间能够由 Nav2 Planner 生成路径。
7. 检查点朝向不会让相机对着墙面过近。
8. `visual_marker` 打卡配置了预期 ID。
9. 视觉任务配置了模型、阈值和失败策略。
10. 路线总长度、预计耗时和最大任务数在限制内。

导航未启动时可进行静态栅格校验；导航已启动时进一步调用 Planner Server 的路径计算接口进行动态校验。

## 7. 巡检任务状态机

### 7.1 状态定义

```text
IDLE                 无任务
PREPARING            加载地图、路线和依赖
LOCALIZING           等待可靠定位
NAVIGATING           正在前往检查点
ARRIVAL_CONFIRMING   Nav2 成功后确认实际位姿和静止状态
CHECKING_IN          执行定位或视觉打卡
CAPTURING            抓拍巡检图片
INSPECTING           执行 YOLO/VLM 检查
RECORDING            持久化检查结果
BLOCKED              持续障碍或导航无进展
RECOVERING           执行有限恢复动作
PAUSING              正在取消 Nav2 并停车
PAUSED_MANUAL        已暂停，允许人工接管
WAITING_OPERATOR     等待操作者决定
ESTOPPED             急停锁定
COMPLETED            任务完成
CANCELLED            任务被取消
FAILED               任务失败
```

### 7.2 主流程

```text
IDLE
 -> PREPARING
 -> LOCALIZING
 -> NAVIGATING
 -> ARRIVAL_CONFIRMING
 -> CHECKING_IN
 -> CAPTURING
 -> INSPECTING
 -> RECORDING
 -> NAVIGATING（下一检查点）
 -> COMPLETED
```

### 7.3 核心状态转换规则

1. `PREPARING -> LOCALIZING`：地图和路线校验通过，Nav2 已激活。
2. `LOCALIZING -> NAVIGATING`：AMCL 和 TF 健康，且收到启动确认。
3. `NAVIGATING -> ARRIVAL_CONFIRMING`：Nav2 Action 返回成功。
4. `NAVIGATING -> RECOVERING`：Action 失败但仍有重试次数。
5. `NAVIGATING -> BLOCKED`：无进展超时或障碍持续存在。
6. `ARRIVAL_CONFIRMING -> CHECKING_IN`：位置、朝向、速度和协方差满足阈值。
7. `CHECKING_IN -> CAPTURING`：打卡成功，且有巡检任务。
8. `CHECKING_IN -> RECORDING`：打卡成功且无巡检任务。
9. `INSPECTING -> RECORDING`：收到合法结构化结果或降级结果。
10. `RECORDING -> NAVIGATING`：结果写入成功并存在下一点。
11. `RECORDING -> COMPLETED`：最后一个点完成。
12. 任意自动状态收到暂停请求后进入 `PAUSING`。
13. 任意运动状态收到有效急停后进入 `ESTOPPED`。

### 7.4 状态持久化

每次状态变化必须在执行下一动作前写入数据库，至少记录：

```text
mission_id
timestamp
previous_state
new_state
checkpoint_id
retry_count
reason_code
detail
robot_pose
```

节点重启后：

1. 不允许自动恢复运动。
2. 将未完成任务恢复为 `WAITING_OPERATOR`。
3. APP 显示上次检查点和中断原因。
4. 操作者选择重试当前点、从下一点继续或结束任务。

8. 取消完成前不允许切换到手动运动。

## 9. 检查点打卡实现

### 9.1 推荐方案：定位围栏 + 视觉标记

打卡采用两层验证：

#### 第一层：定位到达

必须同时满足：

1. Nav2 返回成功。
2. 当前位姿到目标位置的欧氏距离小于 `position_tolerance_m`。
3. 朝向误差小于 `yaw_tolerance_rad`。
4. AMCL 协方差小于配置阈值。
5. 安全仲裁输出速度接近零。
6. 以上条件持续满足 `dwell_sec`。

#### 第二层：视觉标记

1. 相机查找二维码或 AprilTag。
2. 解码出的 ID 与检查点 `expected_marker_id` 完全一致。
3. 连续多帧或多张图片确认，避免单帧误识别。
4. 保存带标记框的证据图。
5. 记录识别时间、位置和重试次数。

### 9.2 QR 与 AprilTag 选择

| 方案 | 优点 | 缺点 | 建议 |
| --- | --- | --- | --- |
| OpenCV QRCodeDetector | 依赖少、二维码可携带文本、APP 易生成 | 远距离和斜视角稳定性一般 | MVP 首选 |
| pyzbar | 使用简单 | Jetson 需要额外系统依赖 | 已有依赖时可用 |
| AprilTag | 角度、距离和弱纹理场景更稳定，可估计位姿 | 需要额外 ROS/算法包 | 可靠版首选 |
| 仅 AMCL 围栏 | 不需张贴标记 | 无法证明到达的是正确物理点 | 只能作为降级方案 |

### 9.3 视觉打卡动作

若标记不在相机中心，可以设计有限扫描动作：

1. 原地保持静止抓拍。
2. 未找到时，安全旋转约 `-15°` 后抓拍。
3. 再旋转约 `+30°` 后抓拍。
4. 最后回到目标朝向。
5. 所有旋转必须走视觉/任务速度通道并经过 `safety_mux`。
6. 人员或近距离障碍出现时立即停止扫描。

扫描角度、次数和速度必须受配置限制，不能由二维码内容或 LLM 动态生成。

## 10. 拍照与智能识别实现

### 10.1 到点抓拍流程

```text
检查点确认到达
 -> 输出零速度
 -> 等待相机自动曝光稳定 0.5～1 秒
 -> 获取 RGB 图像时间戳
 -> 连续抓拍 3 张
 -> 检查模糊度、曝光和图像尺寸
 -> 选择最佳图片并保存原图
 -> 本地 YOLO 推理
 -> 置信度不足时调用 VLM
 -> 校验结构化输出
 -> 保存结果与证据
```

### 10.2 图像质量检查

在送入模型前建议检查：

1. 图像是否为空或尺寸异常。
2. 拉普拉斯方差是否低于模糊阈值。
3. 过暗或过曝像素比例。
4. 图像时间戳是否为检查点到达后。
5. RGB 和 Depth 是否时间接近。

质量不合格时先重新抓拍，不要直接把低质量图片判为“目标不存在”。

### 10.3 本地 YOLO

本地模型负责低延迟、可重复的第一层检测：

1. 检测类别、检测框和置信度。
2. 对多个抓拍结果做投票或最高置信度融合。
3. 输出固定结构，不直接决定运动。
4. 模型不可用时返回明确的 `MODEL_UNAVAILABLE`，不能假装“未检测到目标”。

需要区分“消防栓”和“灭火器”：

1. 通用 COCO 模型可能包含 `fire hydrant`，但对室外、现场角度和光照仍需实测。
2. 室内 `fire extinguisher` 通常需要自定义类别和现场数据训练。
3. 现场还可能需要检测“消防设施被遮挡”“箱门损坏”等状态，这更适合自定义模型或 VLM 复核。

推荐训练流程：

1. 在真实巡检点采集正常、缺失、遮挡、不同光照图片。
2. 划分训练、验证、现场留出测试集。
3. 标注目标框和必要的状态标签。
4. 使用轻量 YOLO 模型训练。
5. 导出适合 Jetson 的 ONNX/TensorRT 模型。
6. 固定模型版本，并在结果中记录模型哈希。

### 10.4 VLM/LLM 复核

VLM 只处理到点静态图片，不处理实时视频。典型用途：

1. 判断门前是否存在目标设施。
2. 判断目标是否明显被遮挡。
3. 对 YOLO 低置信度结果进行语义复核。
4. 生成简短的自然语言说明。

必须要求模型输出固定 JSON：

```json
{
  "task": "fire_extinguisher_presence",
  "present": true,
  "confidence": 0.91,
  "condition": "normal",
  "reason": "门右侧检测到红色灭火器",
  "needs_human_review": false
}
```

小车端必须执行：

1. JSON Schema 校验。
2. 枚举值校验。
3. 置信度范围校验。
4. 请求超时和最大重试限制。
5. 模型异常时的降级处理。
6. 原始响应审计保存，但不能直接当作 ROS 命令执行。

LLM/VLM 不得：

1. 发布 `/cmd_vel`。
2. 修改速度上限。
3. 释放急停。
4. 执行 Shell 命令。
5. 任意访问 ROS Topic。
6. 决定未经白名单配置的恢复动作。

### 10.5 识别结果融合

建议使用确定性规则融合：

```text
YOLO >= 高阈值
    -> 直接采用本地结果

YOLO 在低阈值和高阈值之间
    -> 调用 VLM 复核

YOLO 未检测到且任务为强制项
    -> 使用最佳图片调用 VLM

VLM 超时或输出非法
    -> 标记 UNKNOWN，不得直接标记为“不存在”
```

最终结论至少包含：

```text
PRESENT
ABSENT
ABNORMAL
UNKNOWN
NEEDS_HUMAN_REVIEW
```

### 10.6 证据存储

每个检查点目录示例：

```text
~/.icar/evidence/<mission_id>/<checkpoint_id>/
├── arrival.json
├── checkin_original.jpg
├── checkin_annotated.jpg
├── capture_001.jpg
├── capture_002.jpg
├── capture_003.jpg
├── detection.json
├── vlm_response.json
└── result.json
```

文件数据库记录至少包括：

1. Mission 和 Checkpoint ID。
2. 拍摄时间。
3. `map` 坐标位姿。
4. 图片 SHA-256。
5. 模型名称、版本和阈值。
6. 检测框与置信度。
7. 最终结论。
8. 是否经过人工修改。

## 11. 障碍物与人员阻挡处理

### 11.1 职责划分

| 问题 | 负责模块 | 原则 |
| --- | --- | --- |
| 普通静态/动态障碍 | Nav2 Local Costmap + DWB | 减速、绕行、重新规划 |
| 极近距离障碍 | `safety_mux` 安全门 | 禁止继续前进或立即停车 |
| 人员距离过近 | RGB-D 人员安全节点 | 降速或停车，不主动贴身绕行 |
| 长时间无进展 | Mission Manager | 计时、有限恢复、通知 APP |
| 传感器丢失 | Health Monitor + Safety Mux | 停车并进入故障状态 |

自定义雷达避障节点在导航模式下不应自行选择转向并与 Nav2 抢占控制。建议将其改为只发布：

```text
clear
slow
stop
sensor_fault
```

由 `safety_mux` 对 Nav2 速度做限幅或截断。

### 11.2 障碍物状态分级

#### Level 0：正常

Nav2 正常导航。

#### Level 1：减速区

前方障碍进入减速距离：

1. 限制最大线速度。
2. 允许 Nav2 自主规划。
3. APP 显示黄色提示。

#### Level 2：停车区

障碍进入停车距离：

1. `safety_mux` 输出零速度或阻止正向速度。
2. Mission 状态进入阻挡计时。
3. 障碍离开后采用迟滞阈值恢复，避免频繁抖动。

#### Level 3：持续阻挡

超过配置时间后：

1. 取消当前导航 Goal。
2. 清理局部 Costmap。
3. 等待后重新发送目标。
4. 必要时执行有限旋转或有限后退。
5. 超过最大重试次数后进入 `WAITING_OPERATOR`。

恢复行为必须设定最大距离、最大角度和最大次数。

### 11.3 人员阻挡策略

人员场景比普通障碍更保守：

1. 例如 1.2 m 内限制线速度。
2. 例如 0.55 m 内强制停车。
3. 至少连续 2～3 帧确认后触发，避免单帧噪声。
4. 人员离开后持续安全 1～2 秒再恢复。
5. 不执行朝向人员的自动后退或急转动作。
6. 持续阻挡时可通过蜂鸣器或语音提示，但音量和频率应受限。
7. APP 显示“人员阻挡”，不要笼统显示“导航失败”。

距离阈值必须通过实车低速测试确定，文档中的数值仅是初始建议。

### 11.4 定位丢失

出现以下情况应暂停自动运动：

1. `/amcl_pose` 超时。
2. TF 查询连续失败。
3. 协方差超过阈值。
4. 位姿在短时间内非物理跳变。
5. 雷达与地图匹配明显失败。

处理流程：

```text
发布零速度
 -> 取消 Nav2 Goal
 -> 尝试原地小角度重定位（仅在周围安全时）
 -> 仍失败则通知 APP 设置初始位姿
 -> 人工确认后恢复任务
```

## 12. 控制仲裁与人工接管

### 12.1 建议优先级

```text
1. 锁存急停
2. 传感器故障停车
3. 人员/极近障碍停车
4. 人工接管
5. Nav2 自动导航
6. 视觉跟随或其他自动模式
7. 默认零速度
```

人工接管优先于普通自动任务，但不能绕过急停和近距离安全停车。

### 12.2 接管流程

APP 点击“接管”后：

1. APP 发送 `teleop_acquire`。
2. App Bridge 检查认证和当前控制租约。
3. Mission Manager 进入 `PAUSING`。
4. 取消当前 Nav2 Goal。
5. 等待取消结果或达到安全超时。
6. 发布零速度并确认最终输出为零。
7. `safety_mux` 切换到 `manual`。
8. 创建带有效期的控制租约。
9. 返回接管成功。
10. APP 此后才允许发送摇杆速度。

不能先切到手动再异步取消 Nav2，否则可能短时间出现两个控制源交错。

### 12.3 控制租约

控制租约字段：

```text
lease_id
client_id
issued_at
expires_at
last_heartbeat_at
mode
```

规则：

1. 同时只有一个客户端拥有手动控制权。
2. `move` 命令必须携带有效 `lease_id`。
3. APP 以 10～20 Hz 发送速度，至少以 2 Hz 发送租约心跳。
4. 速度命令超过 300～600 ms 未更新时输出零速度。
5. TCP 断开时立即撤销该连接持有的租约并停车。
6. 新客户端不能静默抢占，除非使用管理员强制接管并产生事件记录。

### 12.4 释放接管和恢复任务

1. APP 先发送零速度。
2. 发送 `teleop_release`。
3. 系统撤销租约并保持停车。
4. 检查 AMCL 和传感器健康状态。
5. APP 显示恢复选项：
   - 重试当前检查点；
   - 从下一检查点继续；
   - 返回指定安全点；
   - 结束任务。
6. 操作者确认后，Mission Manager 再发送新的 Nav2 Goal。

### 12.5 急停与接管区别

| 功能 | 目的 | 自动任务状态 | 如何恢复 |
| --- | --- | --- | --- |
| 暂停 | 暂时停止任务 | 保留上下文 | 显式继续 |
| 人工接管 | 操作者驾驶 | 任务进入 `PAUSED_MANUAL` | 释放租约并选择恢复点 |
| 取消 | 结束本次任务 | `CANCELLED` | 重新创建任务 |
| 急停 | 处理紧急危险 | `ESTOPPED` | 排除危险后显式释放和重新确认 |

## 17. 部署与运行建议

### 17.1 运行模式

建议提供三个明确启动入口：

```bash
# 建图模式
ros2 launch car_bringup mapping_system.launch.py

# 定位和路线编辑验证模式
ros2 launch car_bringup localization_system.launch.py map:=<map.yaml>

# 完整巡检模式
ros2 launch car_bringup patrol_system.launch.py map:=<map.yaml>
```

不要要求操作者组合十多个布尔 Launch 参数。

### 17.2 日志

每次任务建立独立日志目录，并记录：

1. ROS2 节点日志。
2. Mission 状态变化。
3. APP 控制和接管事件。
4. Nav2 目标和结果。
5. 安全触发事件。
6. 模型版本和推理耗时。

### 17.3 进程守护

长期运行时可使用 systemd 启动基础服务，但自动重启后必须保持停车。建议：

1. 底盘和传感器节点可自动重启。
2. Mission Manager 重启后恢复为 `WAITING_OPERATOR`。
3. App Bridge 重启后所有旧控制租约失效。
4. 急停状态持久化或由硬件急停保证。

### 17.4 数据清理

配置证据保留策略：

1. 按天数或磁盘占用上限清理。
2. 优先保留异常任务和人工标记任务。
3. 清理文件前先更新数据库。
4. 路线、地图和报告不可与临时缓存混放。

## 18. 主要风险与应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| Nav2 与安全节点争抢 `/cmd_vel` | 可能绕过急停 | P0 完成唯一速度出口和发布者检查 |
| Foxy 与本地 Humble 行为差异 | 实车构建或接口不一致 | 以 Jetson Foxy 为验收环境，避免使用仅新版本存在的接口 |
| AMCL 在重复走廊定位不稳定 | 错误到点或导航失败 | 增加明显环境特征、合理初始位姿、协方差门槛和二维码复核 |
| 二维码受光照和角度影响 | 打卡失败 | 合理尺寸和安装角度、多帧确认、有限扫描、必要时升级 AprilTag |
| 通用模型不识别室内灭火器 | 巡检结果不可靠 | 采集现场数据训练自定义模型，VLM 仅作复核 |
| 云端 VLM 网络不稳定 | 任务长时间等待 | 超时、重试上限、UNKNOWN 降级和人工复核 |
| 图片占满 Jetson 磁盘 | 系统异常 | 配额、缩略图、自动清理和磁盘健康监测 |
| 多 APP 同时控制 | 控制冲突 | 单一控制租约和审计日志 |
| 任务暂停时 Nav2 尚未取消 | 短暂速度冲突 | `PAUSING` 状态先取消、停车确认，再授予手动租约 |
| 传感器节点假在线 | 安全风险 | 使用消息新鲜度和数据质量，而不是只检查节点名 |

## 19. 推荐的最终演示流程

1. APP 连接小车并展示健康状态。
2. 选择已有地图，显示地图和小车实时位置。
3. 在 APP 打开已保存路线，展示检查点和任务配置。
4. 启动巡检，小车自动前往 CP-01。
5. 到点后识别 CP-01 二维码完成打卡。
6. 自动抓拍并检查消防设施，APP 展示图片和结果。
7. 在下一段路线临时放置障碍，展示停车和重新规划。
8. 让人员靠近，展示减速和安全停车。
9. APP 中途接管小车，手动移动后释放控制权。
10. 选择“重试当前检查点”恢复巡检。
11. 任务结束后打开报告，展示路线、时间、打卡、图片、识别结果和异常事件。

该演示覆盖建图导航、任务编排、视觉识别、安全控制、异常处理、APP 交互和工程证据，能够形成完整而非孤立功能的项目展示。

## 20. 实施结论

当前仓库已经具备完成复杂巡检的主要底层积木，但还没有任务级闭环。最重要的工程方向不是继续扩充简单航点循环，而是增加独立的任务编排层，并确保所有自动和人工速度都经过同一个安全仲裁出口。

推荐严格按以下顺序推进：

```text
修复安全控制链
 -> 实车验证单点 Nav2
 -> 建立路线数据模型和任务状态机
 -> 接入 APP 地图与路线编辑
 -> 实现定位 + 二维码打卡
 -> 实现拍照 + YOLO/VLM 巡检
 -> 完善障碍、人员、定位和网络异常处理
 -> 长时间实车验收
```

完成上述改造后，系统才能从“多个 ROS2 功能节点的集合”升级为“可配置、可观察、可恢复、可人工接管的实际巡检系统”。
