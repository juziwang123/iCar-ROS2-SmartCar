# iCar 智能小车 ROS2 项目设计文档

## 1. 文档目的

本文档用于描述 iCar 智能小车 ROS2 项目的总体架构、模块设计、通信关系、运行流程和扩展思路，为后续代码实现、联调测试和答辩讲解提供设计依据。

## 2. 设计目标

系统设计需要满足以下目标：

1. 模块边界清晰，便于多人并行开发。
2. 接口统一，优先使用 ROS2 标准消息与启动方式。
3. 具备从手动控制到感知、导航、视觉扩展的渐进式开发路径。
4. 兼顾本地开发验证和实车部署适配。

## 3. 总体架构设计

### 3.1 分层结构

系统采用分层设计：

1. 设备层：小车底盘、激光雷达、深度相机、传感器。
2. 驱动接入层：底盘驱动、雷达驱动、相机驱动。
3. ROS2 功能层：控制、避障、跟随、建图、导航、视觉识别。
4. 交互层：键盘控制、APP 控制、远程桌面。
5. 工程支撑层：配置、测试、脚本、文档。

### 3.2 功能架构

```text
APP/键盘/远程控制
        |
        v
  car_app_bridge / car_control
        |
        v
      /cmd_vel
        |
   -----------------------------
   |             |             |
   v             v             v
car_lidar   car_vision   car_navigation
   |             |             |
 /scan         /image       /map /goal_pose
```

系统中的控制链路围绕 /cmd_vel 汇聚，感知链路由雷达和视觉模块提供环境输入，导航模块根据地图和目标点输出自动控制策略。

## 4. 目录与模块设计

### 4.1 顶层目录说明

| 目录 | 设计用途 |
|------|----------|
| src | 存放所有 ROS2 功能包 |
| docs | 存放需求、设计、测试、使用等文档 |
| tests | 存放测试脚本与测试占位内容 |
| scripts | 存放安装、部署、自动化辅助脚本 |

### 4.2 功能包设计

#### car_bringup

职责：

1. 汇总系统启动入口。
2. 管理参数配置。
3. 组织多模块启动顺序。

输入输出：

1. 输入为启动参数和配置文件。
2. 输出为各功能节点的统一运行环境。

#### car_control

职责：

1. 封装底盘运动控制接口。
2. 接收键盘或上位机控制命令。
3. 发布标准速度控制消息。

核心接口：

1. 发布 /cmd_vel。
2. 提供停止、前进、后退、转向等动作封装。

#### car_lidar

职责：

1. 处理 /scan 数据。
2. 实现避障、跟随、警卫等逻辑。
3. 根据规则输出控制建议或告警信息。

设计重点：

1. 识别前向安全距离。
2. 支持阈值参数化。
3. 保留与导航模块协同的能力。

#### car_vision

职责：

1. 接入相机图像流。
2. 实现颜色检测、目标检测与追踪。
3. 输出目标位置或追踪控制建议。

设计重点：

1. 先支持轻量方案，如颜色识别。
2. 再扩展到 YOLO 模型检测。
3. 保证检测输出能够被控制模块消费。

#### car_navigation

职责：

1. 承担建图、定位、导航与巡航任务。
2. 管理地图文件和导航参数。
3. 对接 Nav2 或同类导航组件。

设计重点：

1. 支持地图保存与复用。
2. 支持目标点发布和多点巡航。
3. 与避障策略形成协同而不是互相覆盖。

#### car_app_bridge

职责：

1. 接收 APP 或外部控制端指令。
2. 转换为 ROS2 消息和内部控制逻辑。
3. 为后续状态回传预留接口。

## 5. 通信设计

### 5.1 核心话题

| 话题 | 类型 | 作用 |
|------|------|------|
| /cmd_vel | geometry_msgs/Twist | 小车底盘速度控制 |
| /scan | sensor_msgs/LaserScan | 激光雷达扫描数据 |
| /camera/color/image_raw | sensor_msgs/Image | RGB 图像数据 |
| /camera/depth/image_raw | sensor_msgs/Image | 深度图像数据 |
| /map | nav_msgs/OccupancyGrid | SLAM 地图输出 |
| /odom | nav_msgs/Odometry | 里程计数据 |
| /goal_pose | geometry_msgs/PoseStamped | 导航目标点 |

### 5.2 控制优先级设计

建议采用以下优先级：

1. 急停/停止指令。
2. 安全避障控制。
3. 自动导航控制。
4. 视觉追踪控制。
5. 手动普通控制。

设计原则是任何非安全控制都不能覆盖急停和安全避障逻辑。

## 6. 运行流程设计

### 6.1 开发验证流程

1. 启动本地 ROS2 环境。
2. 构建工作空间并 source。
3. 单独启动目标模块。
4. 使用命令行或可视化工具验证话题与日志。

### 6.2 实车运行流程

1. 小车上电并接入网络。
2. SSH 或 NoMachine 登录小车。
3. 启动底盘基础服务和传感器驱动。
4. 按场景启动避障、建图、导航或视觉模块。
5. 通过手动接管或急停保障现场安全。

## 7. 配置设计

### 7.1 参数配置

建议将以下内容配置化管理：

1. 底盘速度上限。
2. 避障距离阈值。
3. 跟随目标距离。
4. 相机检测阈值。
5. 导航参数与巡航点。

### 7.2 地图与模型资源

1. 地图文件放置在 navigation 模块 maps 目录。
2. YOLO 模型放置在 vision 模块 models 目录。
3. 配置与模型资源应与代码版本保持同步。

## 8. 安全与异常设计

1. 所有自动运动功能必须可人工接管。
2. 模块启动失败时，应尽量保证其他模块不被级联拖垮。
3. 日志输出需要清晰，便于快速定位故障模块。
4. 网络异常时，手册中应提供可回退到命令行调试的方案。

## 9. 可扩展性设计

1. 新增传感器时优先通过标准话题接入。
2. 新增算法模块时尽量复用现有控制接口。
3. 新增演示场景时，优先通过配置文件管理参数差异。
4. 进阶功能如 LLM、多车协同应作为独立子系统接入，而非直接耦合在现有基础控制中。

### 9.1 拓展功能接入位置

1. 自动循线建议作为 car_vision 内的独立节点接入，通过图像偏差计算生成控制建议。
2. 道面缺陷、积水和危险异物检测建议作为 car_vision 的风险检测分支接入，输出检测结果和风险等级。
3. LLM 任务交互建议放在 car_app_bridge 中，将自然语言任务映射为模式切换、导航目标或动作指令。
4. 多车协同建议采用独立命名空间和独立协调节点实现，不直接修改单车控制逻辑。

### 9.2 拓展功能接口约束

1. 所有拓展功能都应通过标准 topic、service 或 action 与主系统交互。
2. 涉及运动控制的拓展功能必须经过统一控制仲裁，不得直接绕过安全控制。
3. 涉及模型推理的拓展功能应保留参数化配置，方便在不同设备环境下调节。
4. 多车协同必须明确主从角色、命名空间和冲突处理策略。

## 10. 当前设计结论

当前项目最合理的设计路径是先把控制、雷达、导航三条主链路做稳定，再将视觉和 APP 交互逐步接入。仓库现有目录划分能够支撑这一策略，后续开发应严格遵循模块边界和接口规范，避免在时间紧张的实训周期内出现大范围返工。
