# iCar 智能小车 ROS2 项目拓展功能实施方案

## 1. 文档目的

本文档用于将项目中的进阶功能从“加分项设想”转化为“可执行任务”，明确拓展功能的目标、优先级、技术路线、分工建议、风险与验收标准，便于项目组在完成主线功能后有序推进。

## 2. 拓展功能范围

结合课程要求与当前项目基础，拓展功能范围包括：

1. 道面缺陷检测与避障。
2. 湿滑积水与危险异物检测。
3. 接入 LLM 实现复杂任务交互。
4. 多车协同与多车队列。
5. 自动循线。

## 3. 总体推进策略

### 3.1 推进原则

1. 主线功能优先，拓展功能不能破坏基础控制、避障和导航链路。
2. 先做低耦合、易演示、能快速出效果的功能，再做高复杂度功能。
3. 每个拓展功能都必须有最小演示版本，不追求一次性做到完整产品级效果。
4. 拓展功能应尽量复用已有雷达、视觉、导航和控制接口。

### 3.2 优先级建议

| 优先级 | 功能 | 原因 |
|--------|------|------|
| P1 | 自动循线 | 视觉实现链路短，易形成单独演示 |
| P1 | 道面缺陷检测与避障 | 与 YOLO/视觉模块复用度高，展示效果强 |
| P2 | 湿滑积水与危险异物检测 | 与目标检测思路一致，可作为缺陷检测扩展 |
| P2 | LLM 复杂任务交互 | 亮点明显，但依赖接口封装和指令设计 |
| P3 | 多车协同与多车队列 | 工程量最大，对设备和时间要求最高 |

## 4. 详细功能实施设计

### 4.1 EF-01 自动循线

#### 目标

基于相机识别地面颜色线或标志线，实现小车沿指定路径自动行驶。

#### 实现思路

1. 使用 RGB 相机获取图像。
2. 采用 HSV 阈值分割或边缘检测提取地面引导线。
3. 计算引导线相对图像中心的偏差。
4. 将偏差转换为角速度控制，发布到 /cmd_vel。

#### 依赖模块

1. car_vision。
2. car_control。

#### 最小可行版本

1. 在固定光照环境下识别单一颜色线。
2. 支持低速直线和简单转向循线。

#### 验收标准

1. 小车能在短距离内沿地面标线连续移动。
2. 偏离线路时能产生可见的修正动作。

### 4.2 EF-02 道面缺陷检测与避障

#### 目标

识别地面裂缝、坑洞或破损区域，并在识别后采取减速、停止或绕行策略。

#### 实现思路

1. 使用相机采集地面图像。
2. 基于 YOLO 或轻量分类方式识别缺陷区域。
3. 将图像中的缺陷位置映射为风险区域判断。
4. 向控制模块发送减速、停止或转向建议。

#### 依赖模块

1. car_vision。
2. car_control。
3. car_navigation 或 car_lidar 作为辅助安全策略。

#### 最小可行版本

1. 能识别预设的缺陷样本图片或地面标记。
2. 检测到缺陷后能触发停车或避让。

#### 验收标准

1. 对预设缺陷目标有稳定检测输出。
2. 在演示场景中能触发对应避险动作。

### 4.3 EF-03 湿滑积水与危险异物检测

#### 目标

识别积水区域、反光湿滑区域或危险异物，并输出提示或控制动作。

#### 实现思路

1. 复用视觉检测模型框架。
2. 使用预定义类别识别积水、异物等风险目标。
3. 根据检测结果输出日志、告警或减速控制。

#### 依赖模块

1. car_vision。
2. car_control。

#### 最小可行版本

1. 能对 1 至 2 类风险目标进行检测。
2. 能以日志、标框或停车形式展示结果。

#### 验收标准

1. 检测结果可视化清晰。
2. 至少对一种风险目标做出控制响应。

### 4.4 EF-04 接入 LLM 实现复杂任务交互

#### 目标

支持用户以自然语言方式向小车下达较高层任务，例如“巡航到指定点”“检测前方目标”“开始警卫模式”等。

#### 实现思路

1. 设计自然语言指令到动作模板的映射。
2. 在 app bridge 或独立模块中增加任务解析节点。
3. 将语言任务转换为 ROS2 控制命令、模式切换命令或导航目标。
4. 输出执行状态，便于演示人机交互闭环。

#### 依赖模块

1. car_app_bridge。
2. car_control。
3. car_navigation。
4. car_lidar 或 car_vision，视具体任务而定。

#### 最小可行版本

1. 支持 3 至 5 条固定模板自然语言指令。
2. 支持文字输入，不强制要求语音输入。

#### 验收标准

1. 用户输入任务后，系统能解析并触发对应动作。
2. 至少完成一次“输入任务-执行动作-反馈结果”的闭环演示。

### 4.5 EF-05 多车协同与多车队列

#### 目标

实现两台及以上小车的基础协同控制，如队列行驶、跟车或同步启动。

#### 实现思路

1. 为不同小车划分命名空间。
2. 设计主车与从车的角色关系。
3. 通过共享目标点、速度指令或队列逻辑实现协同。
4. 增加车间距离或状态同步机制。

#### 依赖模块

1. 多车 ROS2 网络环境。
2. car_control。
3. car_navigation。
4. 可选 car_lidar。

#### 最小可行版本

1. 两台车完成同向启动与停止。
2. 或完成简化版主从跟随演示。

#### 验收标准

1. 两台车能在同一网络下稳定通信。
2. 至少完成一次协同动作演示。

## 5. 架构接入建议

### 5.1 新增模块建议

建议为拓展功能预留如下实现位置：

1. 自动循线可放在 car_vision 中，作为 line_follower.py。
2. 缺陷检测、积水检测、异物检测可归入 car_vision 中，作为 defect_detector.py 或 risk_detector.py。
3. LLM 交互可在 car_app_bridge 中扩展 task_agent.py。
4. 多车协同可在 car_navigation 或新增 coordination 模块中实现。

### 5.2 控制优先级要求

拓展功能接入后，控制优先级仍需遵守：

1. 急停。
2. 安全避障。
3. 自动导航。
4. 拓展功能控制。
5. 普通手动控制。

任何拓展功能都不得跳过安全控制链路直接控制底盘。

## 6. 分工建议

| 功能 | 主负责人 | 协作人 | 主要输出 |
|------|----------|--------|----------|
| 自动循线 | 成员 D | 成员 B | 识别与控制联动演示 |
| 缺陷检测与避障 | 成员 D | 成员 C | 检测输出与避障策略 |
| 湿滑积水/异物检测 | 成员 D | 成员 E | 检测结果、测试记录 |
| LLM 交互 | 成员 A | 成员 B | 文本任务解析与动作触发 |
| 多车协同 | 成员 C | 成员 A、B | 多车通信与队列演示 |

## 7. 时间安排建议

### 7.1 第一阶段

1. 主线功能完成后，先启动自动循线和缺陷检测。
2. 两个功能都可在单车环境内验证，风险较低。

### 7.2 第二阶段

1. 在视觉检测有基本结果后，扩展到积水和异物检测。
2. 同时启动 LLM 文本指令原型。

### 7.3 第三阶段

1. 若设备和时间允许，再尝试多车协同。
2. 若条件不足，可将多车协同保留为设计说明和原型展示。

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 拓展功能挤占主线开发时间 | 主功能不稳定 | 采用阶段门控，主线未完成不进入高阶开发 |
| 视觉模型效果不稳定 | 检测误报漏报 | 先做固定场景和固定类别演示 |
| LLM 输出不可控 | 指令解析错误 | 先限制为模板化命令，不直接开放自由生成控制 |
| 多车资源不足 | 无法实测 | 以命名空间仿真、文档设计或单车模拟替代 |
| 控制冲突 | 小车行为异常 | 严格遵守控制优先级设计 |

## 9. 验收方式

1. 每个拓展功能必须提供至少一种可见结果：运动、检测框、日志、截图或视频。
2. 每个拓展功能必须有负责人和测试记录。
3. 若功能未完全落地，至少应保留设计说明、任务拆分和原型结果，不能只停留在口头设想。

## 10. 结论

拓展功能是项目亮点的重要来源，但前提是必须用可执行方案管理，而不是在主线功能尚未稳定时同时铺开所有方向。对当前项目最合理的做法是优先做自动循线和视觉类风险检测，再按时间推进 LLM 交互和多车协同，这样既能保住主线，也更容易形成答辩亮点。
